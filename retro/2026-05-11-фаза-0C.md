### Обнаруженные расхождения live VST vs docs

**Два новых квирка** (зафиксированы в `plans/01` §7 п.30–31):

1. **`recvWindow` в подписи → `code=100001 "Signature verification failed"`.** Docs-v3 говорят, что `recvWindow` — стандартный параметр в каждом signed-запросе (default 5000ms). На live VST подпись с `recvWindow` (даже = 5000) отвергается. Без `recvWindow` в payload — BingX применяет server-side default и принимает подпись. Нашли smoke'ом на `/user/balance`, `/trade/marginType`, `/trade/leverage`.
2. **BingX подписывает query string в том порядке, как она пришла в URL** — не пересортирует на своей стороне, несмотря на доки. Сначала отправлял `httpx params={"symbol":..., "marginType":..., "timestamp":...}` (Python dict сохраняет insertion order → URL в этом же порядке), а sign_query сортировал по ASCII → разный порядок → reject. Решение: перед отправкой пересоздать dict как `dict(sorted(...))`. Подтверждено на `/trade/marginType`, `/trade/leverage`. `set_position_mode` случайно работал в неотсортированном виде, потому что params было всего 2 (`dualSidePosition` и `timestamp` — `d < t`, естественный alpha-порядок).

Оба квирка покрыты unit-тестами (`test_signed_request_includes_signature_apikey_and_recv_window` — `recvWindow` отсутствует; `test_signed_request_emits_params_in_alpha_sorted_order` — порядок ключей).

## Какую проблему решил

До этой сессии единственный способ узнать «работает ли HMAC-подпись на BingX VST» — попробовать разместить ордер в фазе 0.D, что значит риск ложного fill'а / зависшего ордера на demo + долгая отладка подписи на write-методе. Теперь подпись отлажена на read-методах + idempotent setters, write-операции 0.D начнутся с уверенной подписью.

Дополнительно: аккаунт VST переключён в `ISOLATED` + `one-way` — пререквизиты для ордеров 0.D выполнены. Risk-engine 0.E будет читать баланс/позиции через готовый `PrivateAPI`.

## Решения, принятые на будущее

- **`Settings.env` приоритетнее YAML `config.env`.** Это даёт безопасный путь «положил `BINGX_ENV=vst` → пошёл на VST», без правки `config.yaml` (который остаётся декларативным дефолтом). Гарантия: пока в `.env` нет `BINGX_ENV=live`, никакой signed-запрос не пойдёт на live.
- **`recvWindow` НЕ отправляем.** Поле `signing.recv_window_ms` в YAML оставлено как декларация на случай, если BingX вернёт документированное поведение. В коде не используется. Это нужно держать в голове при отладке: если живой VST вернёт «timestamp expired» — это не значит, что `recvWindow` нужно поднять; это значит, что часы поплыли и нужен `time_syncer.sync()` (что уже делается автоматически).
- **Params в signed-запросе — всегда alpha-sorted dict.** Не передавать «произвольный» dict в `request_signed`. Это инвариант, который держит подпись согласованной с URL.
- **Тестовые ключи только `"test-key"`/`"test-secret"`.** Никаких реальных значений в коде/тестах/моках. Реальный `.env` — только локально.
- **VST-баланс = актив `VST`, не `USDT`.** На demo BingX отдаёт `asset="VST"` для виртуальной валюты. Integration-тест принимает оба варианта (VST на demo, USDT на live).

## Зафиксированные риски (на будущее, до фазы 0.D)

1. **Квирки подписи могут отличаться для write-методов.** Read + idempotent setters работают одинаково (общая логика HMAC). Но для `POST /trade/order` могут быть свои особенности (например, body vs query, формат `attached SL/TP` как stringified JSON — см. plans/01 §7 п.6). Первая задача 0.D — smoke на `place_order` с минимальным валидным набором, чтобы поймать новые квирки на дешёвом сценарии.
2. **Нет фонового time-sync task.** Сейчас `now_ms()` ленив и опционально дёргает sync. Для долгоживущей сессии (24/7 бот) нужен фоновый loop — переедет в 0.D вместе с user-data WS, где уже будет общий event-loop.
3. **`get_fills` лимит `1000` не проверен на live.** Unit-тест ловит локальную валидацию (>1000 → ValueError), но что вернёт BingX на `limit=999` — не выяснено (timeout? пустой массив? часть данных?). Подождём первой реальной истории сделок.
4. **VST окружение нестабильное.** При smoke иногда возможны 5xx — у нас уже стоит retry + backoff. Если на тестах поймаем `code=80012` / `code≈80014` (idempotent setters) — это **успех**, см. `_IDEMPOTENT_OK_CODES` в `private.py`. Любой другой код = реальный баг.
5. **Утечка `signature` в логи.** Сейчас в коде нет логирования signed-запросов. Когда добавим в 0.E — маскировать `signature` и `X-BX-APIKEY`. Сейчас риск нулевой, потому что логов нет.
