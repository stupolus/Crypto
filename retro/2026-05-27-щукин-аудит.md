## II. Глубокий аудит кода — конкретные находки

### 🔴 Находка 1 — funding_short_block почти наверняка off by 10–100×

`config.yaml:` `funding_short_block: -0.015` (−1.5%).

`strategy.py:295-300`:
```python
def _funding_allows_short(self, ts: int) -> bool:
    funding = self._funding.get_funding_rate(self._cfg.symbol, ts)
    if funding is None:
        return True
    return funding > Decimal(str(self._cfg.funding_short_block))
```

**Проблема:** на крипто-перпах funding идёт 8-часовыми периодами,
типичный диапазон ±0.01% (≈ ±0.0001). Порог **−1.5% = −0.015 — это
~150 SD от среднего**, никогда не срабатывает. Фильтр фактически
мёртвый.

**Источник ошибки:** в DOLF-плане 23 цитата из статьи Щукина «funding
≤ −1%» — это **кумулятивный funding** (за сутки / за период), не один
тик. Реализация интерпретировала как порог одного значения. Это **прямая
ошибка переноса методологии в код**.

**Решение:** не фикшу — стратегия всё равно мёртвая (edge не
подтверждён). Но фиксирую как **урок** для будущих DOLF-исследований:
- funding-пороги указывать **за период** (per-8h tick) или **накопленные
  за окно** (Σ N последних тиков) — никогда не путать.
- При парсинге методологий из источников **уточнять единицы**.

### 🔴 Находка 2 — реализовано только 2 из 11 сетапов DOLF

`plans/23-dolf-формализация.md` формализует **11 сетапов**:
- LONG: L1, L2 (золотой funding-сетап), L3, L4, L5, L6.
- SHORT: S1, S2, S3, S4, S5.

`strategy.py` реализует только **A1** (≈ L6: long-liq sweep на
экстремуме) и **A2** (≈ S5: short-liq sweep). Остальные 9 сетапов
методологии — **не реализованы** в этом коде.

**Что это значит для аудита:** название «методология Щукина» в
docstring **вводит в заблуждение**. Реализована не методология, а
2 сетапа на ликвидационных каскадах. То, что эти 2 не дали edge, не
закрывает всю методологию — закрывает 2/11.

**Однако** план 23 закрыл методологию **в целом** через `dolf_matrix.py`
(прямой бэктест 10 детекторов на сырых данных, без этой стратегии-обёртки).
Так что общий вердикт «edge не подтверждён» остаётся.

### 🟡 Находка 3 — time-aliasing sweep-детекции на 4h+

`_detect_sweep` берёт `bucket` ликвидаций на одном `ts`. На таймфрейме
**4h/1d** в один bucket попадают часы/сутки реальных каскадов. Каскад,
длившийся 30 минут 8 часов назад, и каскад прямо сейчас неразличимы
в 4h-bucket.

В Щукинской механике важна **последовательность**: каскад → стабилизация
→ вход. Тут стабилизация моделируется как `cycle_wait_bars=2` (= 8h на
4h-фрейме). Это **жёсткий искусственный интервал**, не «cycle complete»
в смысле падение интенсивности ликвидаций.

**Это не баг кода, это органическое ограничение таймфрейма.** На 4h+
DOLF в принципе не тестируется честно. На 1h+ нужен Coinglass STANDARD
($59+/мес).

### 🟡 Находка 4 — CVD-логика как «net change over N bars» очень слабая

`_cvd_confirms`:
```python
change = cvd[-1] - cvd[0]
return change > 0 if side == "BUY" else change < 0
```

Это просто знак суммы дельты за N баров. Без нормализации на цену/объём,
без сравнения с baseline. На больших ТФ CVD за 6 баров — почти случайный
sign-flip.

В Щукинской механике CVD используется как **подтверждение направления
с микроструктуры** — а не как сумма N свечей. На 4h+ CVD как
самостоятельный сигнал бессмысленен.

Плюс: если provider пуст (Coinglass HOBBYIST не отдаёт CVD на части
символов / на 4h интервале), `cvd[]` пустой → `False` → стратегия
**молчит**. Это объясняет, почему живой прогон вообще не торговал у
владельца — она не могла подтвердить CVD на доступных данных.

### 🟢 Находка 5 — анти-look-ahead через провайдеры — корректно

`StaticLiquidationProvider`/`StaticOpenInterestProvider`/etc. отдают
данные только `≤ ts`. Это **главный механизм**, который защищает
бэктест от подсматривания будущего. Проверено в `core/signals/tests/`.

Это **рабочая инфраструктура**, которую стоит сохранить вне зависимости
от судьбы этой стратегии.

### 🟢 Находка 6 — структура composite + protocol-совместимость

`LiquidationReversalStrategy` реализует `Strategy` protocol — единый
код для backtest и live. Это **архитектурно правильно** и должно быть
образцом для других composite-стратегий.

Сама обвязка (`on_candle_close` → setup → cycle_wait → confirm) — чистый
state-machine. Тесты (7 unit-тестов) покрывают ключевые ветви.

### 🟢 Находка 7 — `dolf_matrix.py` / `l3_killtest.py` — методологически
ценные

Это не сама стратегия, но **методология её проверки**:
- `dolf_matrix.py` — параметризованный прогон 10 детекторов × N монет,
  IS/OOS-сплит, итоговая матрица.
- `l3_killtest.py` — kill-тесты (издержки, WF, overlap-корр., coin-
  consistency).

Эти скрипты — **рабочий шаблон** для любого будущего DOLF-like edge.
Их выводы (overlap-корр. убил portfolio-edge, coin-consistency убил
ETH) — образцовые. Удалять или прятать не нужно.

## VI. Что делать с DOLF в будущем

Если когда-то решим вернуться:

1. **Сменить таймфрейм** на 15m/1h — нужен Coinglass STANDARD ($59+/мес).
   На 4h+ методология тестируется органически плохо (Находка 3).
2. **Пересмотреть funding-порог** как cumulative-за-окно или per-tick с
   правильными порядками (Находка 1).
3. **Реализовать недостающие 9 сетапов** (L1-L5, S1-S4) — план 23
   уже их формализовал, можно как отдельные детекторы поверх тех же
   примитивов.
4. **Прогнать через `dolf_matrix.py` + `l3_killtest.py`** — методология
   проверки готова.
5. Если что-то выживает kill-тесты — отдельный план + явное снятие
   live-guard.

До тех пор — закрыто.
