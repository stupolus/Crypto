Исполнитель прогнан против РЕАЛЬНОГО BingX VST (демо-деньги) напрямую
через HTTPS из dev-среды (исходящий SSH к VPS закрыт firewall'ом —

## Баги, найденные живым прогоном (только live бы поймал)

1. **read-error crash (#171):** APIError BingX (100410 rate-limit) на
   get_balance/get_positions ронял `_rebalance` трейсбеком. Фикс:
   graceful ABORT без записи state → идемпотентный ретрай.
2. **hedge reduceOnly (#172):** close_position в hedge слал reduceOnly
   → BingX 109400. Close-leg ребаланса падал на TradFi-перпах;
   kill-switch на них тоже бы падал. Фикс: reduceOnly только в
   one-way; в hedge опускается (флаг closes_position).

Оба прошли бы юнит-тесты с фейковым API — нашёл только реальный VST.

## Находка для решения владельца (НЕ баг)

CL (нефть) LONG отклонён: стоп Faber ~26% (широкий), консервативная
пред-входная liq-оценка при потолке 3x даёт буфер < требуемого 30%
расстояния стопа. Реальный risk-based размер крошечный → реальная liq
далеко, но проверка берёт 3x-cap. Итог: ¾-аллокация при широком стопе.
RiskEngine FIXED — не трогал. Решение: (A) принять ¾, либо (B) отдельным
планом пересмотреть liq-оценку на risk-based плечо.

## Вывод

  (после 2 фиксов). Бот корректно считает сигнал, входит, держит,
  реконсилирует в hedge без дублей.
- **Критерий 1 (4-недельный таймер) — ОТКРЫТ:** нужен персистентный

- **Критерий 4 — с оговоркой:** ¾ вместо ⁴⁄₄ из-за liq-buffer на нефти
  (поведение корректное, требует решения A/B).
- **PnL: НЕ оценивается** — 4 недели = ~1 ребаланс, статистически
  недоказательно.
