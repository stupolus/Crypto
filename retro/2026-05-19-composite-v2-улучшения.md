# Retro 2026-05-19 — composite v2: 3 улучшения + сравнение

План 43. Запрос: «улучшить стратегию 3 раза». Сделано как отдельный
кандидат v2, **демо (v1) не тронут** (pre-registered эксперимент).

## 3 структурных улучшения (opt-in, default = v1)

1. **Confidence-gate** (`min_confidence`): отсекаем слабый консенсус —
   агрегатор уже считал `confidence_raw`, стратегия игнорировала.
2. **Адаптивный TP R** (`tp1_r_adaptive`): цель шире при сильном
   сигнале (1.0R→2.5R по confidence) вместо хрупкого фикс-1.5R.
3. **ATR-percentile режим-фильтр** (`atr_pct_min/max`): не торгуем
   мёртвый чоп / блоу-офф — у composite не было рыночного контекста.

Все три — opt-in с backward-safe дефолтами. `config.yaml`/
`config-4h-demo.yaml` (демо) не изменены — доказано тестом
`test_v2_defaults_preserve_v1` + все прежние тесты зелёные. v2 живёт
в `config-v2.yaml`.

## Честный вывод

- v2 режет сделки (фильтры работают как задумано), убыток по модулю
  меньше, OOS PF выше (0.61 vs 0.49) — **«менее плохо», но не плюс**.
- Обе версии убыточны. Выборка 2–7 сделок = статистически ноль
  (урок iter#3/D3). Делать вывод «v2 лучше» НЕЛЬЗЯ — это шум.
- Улучшения теоретически верны и повышают робастность (меньше слабых
  входов, контекст волатильности), но это **не превращает composite
  в edge**. Натягивать «убыток вдвое меньше» в успех — запрещено.
- Тариф Coinglass (4h/~6мес/429) полноценный WF не даёт — честный
  вердикт по v2 невозможен здесь, как и по v1.

## Решения

- v2 — **не в live**. Тот же gate, что v1: только после прохождения
  критерия iter#4 на нормальных данных (апгрейд тарифа Coinglass).
- Демо v1 продолжает идти по плану 31 без изменений.
- Параметры v2 — pre-registered из теории, НЕ подгонялись и не будут.
- Дальнейшее «улучшение» без нормальных данных = угадывание; стоп
  до апгрейда тарифа ИЛИ результата forward-демо v1.

Прод/core/risk/демо не тронуты.
