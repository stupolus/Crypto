# Ретроспектива: 2026-05-11 — фаза 0.E (hardening BingX-адаптера)

Дата: 2026-05-11. Пятая сессия дня после [0.D part 2](2026-05-11-фаза-0D-part2.md).
Тема: persistence + reconcile + метрики + маскирование секретов + security review.

## Что было

После 0.D part 2: ордера + user-stream + compensating-close работают. Но к live не готовы:
- Рестарт = потеря in-memory ack-кэша.
- WS-разрыв = пропуск событий, без synthetic sync.
- Логи signed-запросов не защищены от утечки `signature`/`X-BX-APIKEY`.
- Нет baseline-метрик latency/slippage — не на что ориентироваться при переходе на live.

## Что стало

### Новые модули
- **`adapters/bingx/journal.py`** — `OrderJournal` на SQLite (stdlib). Журналирует `pending → acked → filled / canceled / failed`. PK = `client_order_id` (наш UUID). Async API через `asyncio.to_thread`. При старте процесса страт читает `list_pending(symbol)` чтобы увидеть зависшие отправки.
- **`adapters/bingx/metrics.py`** — `MetricsWriter` (JSON-lines), `OrderMetric`, `compute_slippage_bps`. Латенси = `ack_received_ms − request_started_ms`. Slippage в bps относительно mark price на момент отправки (стратегия передаёт через `request_mark_price`).
- **`UserStreamReconcileEvent`** в `private_models.py` — синтетическое событие, эмитируется адаптером после успешного WS-коннекта с REST-снимком `balance + positions + open_orders`.

### Изменения
- `BingXUserDataStream._run_session` — после connect эмитит RECONCILE первым, до WS-событий.
- `BingXClient` — `mask_signed_url(url)` и `mask_headers(headers)`. Заменяют значения `signature` и `X-BX-APIKEY` на `***`. Готово к подключению при добавлении логов запросов.
- `PrivateAPI.__init__` принимает опциональные `journal=`, `metrics=`. Без них работает как раньше.
- `PrivateAPI.place_order(req, *, request_mark_price=None)` — записывает в journal `pending` до отправки, `acked` после успешного ack, `failed` при transport/compensating-close. Метрики пишутся в JSON-lines.

### Тесты
- 11 новых unit (`test_journal.py`, `test_metrics.py`, `test_masking.py`): CRUD + status transitions + slippage edge cases + masking.
- 105 unit зелёные, coverage ≥70%, ruff + mypy strict — чисто.
- Integration на live VST: reconcile-event приходит первым; journal-запись `pending → acked` проверена.

## Какую проблему решил

До 0.E адаптер «выглядел» рабочим, но **не годился для live**:
- При случайном падении процесса между `place_order` и push-событием — следующий запуск задублил бы ордер.
- При WS-разрыве на 30 секунд — все события потерялись, стратегия не знала бы текущего состояния.
- Latency/slippage никто не измерял — оптимизировать «выглядит быстро».

После 0.E:
- **Persistence:** SQLite журнал переживает рестарт. Стратегия при старте видит, что было неподтверждено.
- **Reconcile:** push-стрим всегда начинается с RECONCILE-снимка. Стратегия может строить state без «дыр».
- **Метрики:** `ops/metrics.jsonl` накопит baseline до фазы 1 → есть на что ссылаться.
- **Безопасность:** хелперы маскирования готовы для логирования (когда подключим).

## Что НЕ сделано / отложено

- **Telegram-алерты** — отдельный план (нужен бот, токен). На MVP пишу алерты в stderr/journal.
- **Стандалоновые stop_market / stop_limit / tp_market ордера** — отложены до момента, когда стратегия попросит.
- **Парсеры TG / Hyperliquid** — отдельный план, не блокер для первой стратегии.
- **Backtest engine** — план перед стратегией.
- **Reconcile при потере push-события (не reconnect)** — текущий reconcile срабатывает только на reconnect. Если событие потерялось «в полёте» — это не покрывается. Решение: периодический soft-reconcile (раз в N минут) — задача фазы 1.

## Решения, принятые на будущее

- **SQLite без ORM.** Достаточно для одного процесса, отсутствие зависимостей делает journal предсказуемым. Если потребуется multi-process — переедем на DuckDB / Postgres, но пока YAGNI.
- **`OrderJournal` async через `asyncio.to_thread`.** sqlite3 sync, но мы не хотим блокировать event loop. Альтернатива — `aiosqlite`, но это новая зависимость без явной выгоды.
- **Metrics через JSON-lines, не CSV.** Decimal сериализуется как строка, нет проблемы с локалью/separator. Парсинг через `jq` / pandas прямой.
- **`PrivateAPI(journal=, metrics=)` опциональные.** Тесты и smoke без persistence; live и долгие demo — с обоими. По умолчанию `None` чтобы не создавать SQLite в неожиданных местах.
- **`UserStreamReconcileEvent.event_type = "RECONCILE"`.** Строкой, не enum'ом — `Literal["RECONCILE"]`. Совместимо с union `UserStreamEvent`, ясно различимо в стратегии.
- **Reconcile эмитим только на successful connect, не на каждый поллинг.** При отсутствии push-разрыва RECONCILE приходит только один раз — на старте. Это упрощает state-машину стратегии.

## Зафиксированные риски (на будущее, до фазы 1)

1. **Periodic soft-reconcile отсутствует.** Если push-событие потерялось «между» reconnect'ами (например, BingX баг) — мы об этом не узнаем. На фазе 1 добавить таймер каждые 30 сек: дёргать `get_open_orders` и сверять с journal.
2. **`ops/metrics.jsonl` не ротейтится.** Файл будет расти. На фазе 0.E это OK (низкая частота), но на live при 100+ ордерах в день нужна ротация (daily).
3. **`OrderJournal` SQLite на default `journal_mode=DELETE`** — нет WAL. При параллельных writers возможны блокировки. Для single-process — не проблема, но если запустим параллельно `risk-engine` который тоже пишет — нужен WAL (`PRAGMA journal_mode=WAL`).
4. **Маскирование не покрывает body запросов (только URL + headers).** POST `/trade/order` шлёт параметры в query, не в body — поэтому body всегда пуст. Если BingX когда-то переедет на body — нужно расширить.
