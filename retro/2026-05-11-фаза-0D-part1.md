# Ретроспектива: 2026-05-11 — фаза 0.D part 1 (первые ордера на VST)

Дата: 2026-05-11. Третья сессия дня после [фазы 0.C](2026-05-11-фаза-0C.md).
Тема: `place_order` с attached SL, `cancel_order`/`cancel_all`, `close_position` (kill switch).

## Что было

После 0.C: приватный read + idempotent setters работают на VST, аккаунт настроен (ISOLATED, one-way, 3x leverage), но ни одного write-метода для ордеров. Пользователь пополнил VST-баланс до 100 000 VST.

## Что стало

### Новые модели
- `OrderRequest` — наша доменная модель. Жёсткие валидаторы:
  - `entry` без `attached_stop_loss` → `ValueError` (инвариант риск-профиля).
  - `reduce_only` с `attached_stop_loss/take_profit` → `ValueError`.
  - `LIMIT` без `price` → `ValueError`.
  - `MARKET` с `price` → `ValueError`.
  - `quantity ≤ 0` → `ValueError`.
- `OrderAck` — короткий ответ `POST/DELETE /trade/order`. Отличается от `Order` (для openOrders): меньше обязательных полей, `orderId` приходит `int` (snowflake) → каст в `str` единого контракта.

### Новые методы `PrivateAPI`
- `place_order(req)` — атомарный entry+SL/TP через stringified JSON в `stopLoss`/`takeProfit`. `workingType=MARK_PRICE` фиксирован (защита от stop-hunt). Авто-генерация `client_order_id` если не задан.
- `cancel_order(symbol, order_id|client_order_id)` — DELETE с маршрутом по любому из двух id.
- `cancel_all(symbol)` — DELETE всех открытых; idempotent при «nothing to cancel» (code 80018/80020 или строки-подсказки).
- `close_position(symbol)` — kill switch: `cancel_all` → `get_positions` → market reduce_only в обратную сторону. Принудительно `position_side="BOTH"` (one-way инвариант). Idempotent на flat-аккаунте (`None`).

### Тесты
- 11 новых unit (валидация OrderRequest, JSON-сериализация stopLoss/takeProfit, cancel_order edge cases, close_position flat/long).
- Integration на live VST: 1 полный smoke (open market+SL on BTC-USDT 0.0001 → assert position → close → assert flat). Прошёл с первого захода после фиксов.
- Покрытие: **83.84%** (target ≥70%). 83 unit + 11 integration = 94 теста, все зелёные.
- ruff + mypy strict — чисто на 23 source-файлах.

### Обнаруженные расхождения live VST vs docs (квирки 32–33 в plans/01 §7)

1. **`stopPrice` обязан быть JSON-числом, не строкой.** В первом прогоне отправил `"stopPrice":"80499.4"` → `code=109400 "Mismatch type float64 with value string"`. Docs-v3 показывают пример без кавычек (`stopPrice:31968.0`), но это легко прочитать как «JSON в строке, значения как угодно». Фикс: `_attached_protective` сериализует `stopPrice` как `float(decimal)`.
2. **В one-way write-методах `positionSide` обязан быть `"BOTH"`.** `GET /user/positions` для one-way отдаёт `positionSide="LONG"`/`"SHORT"` (видимо, как сторона позиции для удобства), а `POST /trade/order` такие значения отвергает с `code=109400 "...can only be set to BOTH"`. Фикс: `close_position` принудительно ставит `"BOTH"`.

Оба квирка зафиксированы в `plans/01` §7 п.32–33.

## Какую проблему решил

До этой сессии вся торговая часть существовала только в декларациях. Теперь:
- Адаптер технически может открыть и закрыть позицию на VST по нашим инвариантам.
- Инвариант «нет позиции без SL» вкомпилирован на уровне модели (`OrderRequest` бракует entry без stop) — стратегии физически не смогут отправить «голый» entry.
- Kill switch работает: одна команда `close_position(symbol)` снимает все защитные ордера и закрывает позицию.

## Решения, принятые на будущее

- **`OrderRequest` инвариант «entry без SL = ValueError» — на уровне модели, не на уровне адаптера.** Любой код, который попытается собрать такой запрос, упадёт ещё до сети. Это сильнее «адаптер проверяет перед отправкой» — нельзя случайно обойти.
- **`OrderAck` ≠ `Order`.** Live VST показал, что POST/DELETE возвращают уплощённый ответ с другим набором полей. Соблазн «сделать `Order` всеополечным» приводит к проблеме «модель ничего не гарантирует». Лучше две модели с чёткими контрактами.
- **`stopPrice` — `float`, не `Decimal`.** BingX внутри использует float64, и serialization-fence на нашей стороне до запроса. Точность symbol = 1 digit для BTC, float достаточно. Стратегия округляет под precision контракта **до** создания `OrderRequest`.
- **`position_side="BOTH"` принудительно в `close_position`.** Это инвариант one-way режима, не «иногда LONG, иногда SHORT». Адаптер не доверяет ответу `get_positions` в этой части.

## Зафиксированные риски (на будущее, до 0.D part 2)

1. **`place_order` не имеет compensating-close fallback'а.** Если по какой-то причине BingX вернёт ack без подтверждённого SL (квирки, частичный сбой) — позиция останется без стопа. Сейчас полагаемся на атомарность attached SL/TP, что на VST подтвердилось. Compensating-close — обязательная задача 0.D part 2 (перед live).
2. **Нет таймаута на ack.** Если BingX начнёт отвечать > 10 сек, наш timeout (read=10s по YAML) сработает и retry-policy может попытаться. Для POST `/trade/order` retry **запрещён** в `_send_with_retry` (метод не идемпотентен без `clientOrderID` ack-кэша). Сейчас идемпотентность держится через уникальный `client_order_id` на стороне BingX, но мы не проверяем, что повторный POST с тем же coid не создаст дубликат — это задача 0.D part 2.
3. **Только market-ордера протестированы на VST.** Limit с attached SL/TP — только unit-тестом. Smoke на limit — добавить в 0.D part 2.
4. **`cancel_all` коды «nothing to cancel» не верифицированы на VST.** Решение `_is_nothing_to_cancel` глотает 80018/80020 — но они угаданы, не подтверждены. Если в 0.D part 2 будет реальный сценарий «отменить пустой список» — проверить точный код от BingX и зафиксировать в plans/01 §7.

## Итог

Полный жизненный цикл позиции на BingX VST работает. Адаптер физически не позволяет отправить entry без SL. Следующий шаг — стабильность (user-data stream + reconcile + dead-man timer), и только после этого — план первой стратегии.
