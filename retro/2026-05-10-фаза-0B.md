# Ретроспектива: 2026-05-10 — фаза 0.B (первый код в проекте)

Дата: 2026-05-10. Вторая сессия дня после [VPS-инфраструктуры](2026-05-10.md).
Тема: каркас BingX-адаптера + публичные REST + WS market data.

## Что было

После закрытия фазы 0.A (аудит docs-v3) в репозитории не было ни одного `.py`.
План `plans/02-фаза-0B.md` определял задачи сессии: структура `adapters/bingx/`,
HTTP-клиент с подписью / rate-limit / retry, четыре публичных REST-метода,
WS-каркас на market data + 70%+ покрытие тестами.

Особенность: первый код, поэтому в нагрузку — выбор стека (Python 3.13, ruff,
mypy strict, pytest), `pyproject.toml`, venv, выбор клиентских библиотек
(httpx + websockets + pydantic v2 + pydantic-settings).

## Что стало

### Инфра разработки

- `python@3.13` поставлен через brew (Mac локально). VPS уже на 3.13.
- `venv/` в worktree (gitignored).
- `pyproject.toml`: editable install + dev-extras, конфиги ruff (line-length 100,
  отключены RUF001-003 как false-positives для русского), mypy strict, pytest +
  pytest-asyncio + pytest-cov + respx, coverage `fail_under = 70`.

### Адаптер (8 модулей + тесты)

- `adapters/bingx/config.yaml` — все числа: URLs (live + VST + backup), HTTP-таймауты
  и retry, rate limits (350/10s = 70% от 500/10s официальных), recvWindow,
  REST-эндпоинты, klines (limit_max=1440), WS-параметры (Ping/Pong, watchdog 30s,
  reconnect backoff), defaults для smoke, invariants (one-way / ISOLATED).
  Каждое число со ссылкой на источник.
- `adapters/bingx/config.py` — pydantic-валидация YAML с `extra="forbid"`,
  свойство `active_rest_base` / `active_ws_market` по env.
- `adapters/bingx/exceptions.py` — иерархия `BingXError` → `Network/RateLimited/
  Auth/Server/APIError/InvalidResponse/WebSocket/ConfigError`. Каждый класс с
  обоснованием в docstring (когда поднимаем, что делать).
- `adapters/bingx/models.py` — pydantic-модели `ServerTime`, `Contract`, `Ticker`,
  `Kline`. Все денежные поля `Decimal`. Валидатор дефиса в `symbol`. `extra="ignore"`
  чтобы новые поля от BingX не ломали парсинг.
- `adapters/bingx/client.py` — `BingXClient` (async httpx), `sign_query` (HMAC-SHA256,
  сортировка по ASCII, без URL-encoding в подписной строке), `_TokenBucket` (sliding
  window, не классический bucket). `request_public` и `request_signed` (заготовка —
  без ключей бросает `AuthError`).
- `adapters/bingx/public.py` — `PublicAPI`: `get_server_time`, `get_contracts`,
  `get_contract`, `get_ticker`, `get_klines`. Нормализация символа `BTCUSDT` → `BTC-USDT`.
  Локальная валидация `limit ≤ 1440`. Сортировка свечей ASC.
- `adapters/bingx/websocket.py` — `BingXMarketWebSocket` с session-loop, прозрачным
  авто-реконнектом и переподпиской, gzip-декомпрессией, текстовым Ping/Pong,
  async-iterator API через `asyncio.Queue` на канал. Watchdog 30s. DI через
  `connect_factory` — позволяет тестам подменять фейковый ClientConnection.

### Тесты

- **39 unit-тестов** (respx-моки), покрытие **83.02%** (target ≥70%).
- **4 integration-теста** против живого BingX (только public): server_time,
  contract BTC-USDT с подтверждением `tradeMinUSDT == 2`, klines ASC,
  WS-подписка на `BTC-USDT@kline_1m`.
- ruff + mypy strict — чистые на 13 source-файлах.

### Обнаруженные расхождения live BingX vs docs-v3 (новые квирки §7 п.27–29)

Integration-тесты сразу нашли три расхождения с docs:

1. **WS-формат интервала**: docs говорят `BTC-USDT@kline_1min`, live принимает
   только `BTC-USDT@kline_1m` (REST-форму). Альтернатива даёт `code=80015`.
2. **`/quote/contracts` не отдаёт `maxLongLeverage` / `maxShortLeverage`** в
   live — только в примере docs. Реальное плечо берётся через
   `POST /trade/leverage` (фаза 0.C).
3. **Klines возвращаются DESC** (newest first), не ASC. Адаптер сортирует ASC
   локально — стратегии/бэктест получают удобный порядок.

Все три зафиксированы в `plans/01` §7. Без integration-тестов мы бы это словили
уже в момент сборки стратегии.

## Какую проблему решил

До этой сессии мы знали API в теории. Теперь у нас:
- **Работающий публичный канал к BingX** с типизированными моделями и тестами.
- **Адресованы 3 расхождения docs vs live** — каждое стоило бы отдельной
  отладочной сессии в фазе 1.
- **Архитектурно изолированный HTTP-транспорт** (`client.py`) от логики
  методов (`public.py`) — фаза 0.C добавит signed-методы без переписывания.
- **WS-каркас с auto-reconnect** — потребитель не теряет стрим при разрывах,
  это критично для kill-switch инвариантов risk-движка.

## Решения, принятые на будущее

- **`Decimal` для всех денежных полей, никогда `float`.** Это особенно важно
  для размера позиции и расчётов риска (см. `бизнес/риск-профиль.md`).
  Pydantic парсит Decimal из str/int/float корректно.
- **`extra="ignore"` в pydantic-моделях.** BingX добавляет поля без анонса
  (как мы увидели с `size`/`launchTime`/`displayName` в `/contracts`). Игнорим
  новые поля, не падаем.
- **Token-bucket на 70% от официального лимита.** Резерв на всплески и
  параллельные запросы.
- **Integration-тесты с маркером `integration`** — не в CI по умолчанию,
  запуск руками `pytest -m integration`. Защита от того, чтобы CI зависел
  от внешнего сервиса.
- **Авто-реконнект WS прозрачный для потребителя.** `subscribe()` отдаёт
  async-iterator; сессионный цикл переустанавливает коннект и переподписку
  без участия пользователя.

## Зафиксированные риски (на будущее, до live)

1. **`websockets` 14.x deprecation warnings** при доступе к `ConnectionClosed.code/.reason` —
   уже учтены через `e.rcvd.code`. При апгрейде до 15.x проверить, не сломалось ли.
2. **Подпись HMAC проверена только юнит-тестом, не на VST.** Smoke на VST — первая
   задача фазы 0.C.
3. **Локально на Mac тестируется, на VPS пока не запускался.** При первом
   запуске на VPS проверить, что venv создаётся и зависимости встают (Python 3.13
   там уже есть).
