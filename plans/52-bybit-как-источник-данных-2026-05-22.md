# План 52 — Bybit как параллельный источник klines (не trading)

## Дата: 2026-05-22 · Запрос пользователя: Bybit для анализа + demo/live

## Решение (по дереву goal-команды — сам, без вопросов)

Запрос делится на два разных коммита по объёму/риску:

- **Klines/данные** (this plan): public V5 API, без auth, без денег.
  Малая инфра, юнит-тестируется без сети. **Делаю сейчас.**
- **Демо/live торговля** (полный `adapters/bybit/` как у BingX):
  signed-API, ордеры, websocket, journal/metrics — это **недели
  работы**. Текущий вердикт — NO-GO (валидируемого edge нет ни на
  BingX, ни в info-edge probe). Строить полный адаптер для второго
  венчура когда нечего деплоить = ровно «метания», которые goal-
  команда запретила. **Откладываю до validated edge** (composite
  forward-демо положительный ИЛИ платный probe найдёт сигнал).

## Что сделано (klines)

`scripts/download_klines_bybit.py`: Bybit V5 `/v5/market/kline`,
public, без auth. Категории linear/inverse/spot. Интервалы V5
(1/3/5/15/30/60/120/240/360/720/D/W/M). Пагинация по окнам, щадящий
sleep против rate-limit. Slug совместим со схемой BingX-файлов
(`<sym>-usdt-<iv>m.jsonl`), формат строк = `adapters.bingx.models.Kline`,
поэтому `run_backtest` / `walk_forward` читают Bybit-данные **без
правок** — единый бэктест-харнес на двух источниках.

4 юнит-теста на pure-парсере (без сети): slug, парсинг и сортировка,
пустой ответ, BingX-Kline совместимость.

## Сетевой блокер этой сессии (зафиксировано честно)

Cloud-IP контейнера на стороне Bybit edge → **403 даже на public**
endpoint (`/v5/market/time`, все 3 хоста: api.bybit, api.bytick,
api-testnet). Скачивание данных из этой сессии **невозможно**.

Скрипт запускается с **VPS (Hostinger)** или локально — там Bybit
доступен. Не блокер проекта — блокер моего sandbox.

## Команды (для запуска на VPS/локально)

```
.venv/bin/python -m scripts.download_klines_bybit \
    --symbol BTCUSDT --interval 15 --months 24
.venv/bin/python -m scripts.download_klines_bybit \
    --symbol ETHUSDT --interval 60 --months 24
```

Результат: `data/candles/btc-usdt-15m.jsonl`, `eth-usdt-60m.jsonl` —
сразу читаются `walk_forward`/`run_backtest`. Если у Bybit история
глубже BingX, можно прогнать те же стратегии на бОльшем sample и
получить более надёжный WF (формальная re-проверка NO-GO при росте
выборки = разрешена по goal-команде, это **тот же эксперимент**, не
новая гипотеза).

## Что НЕ делаю

- Изменение существующего BingX-runner'а или прода.

  нужны — public). Ключи остаются в env пользователя, в репо не
  попадают; пригодятся при будущей build-фазе trading-адаптера.
