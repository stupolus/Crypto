# План 53 — markitdown как единый ридер PDF (2026-06-06)

## Первичный замысел

Сейчас PDF из `бизнес/inbox/pdf/` читаются «как есть» (Claude открывает файл напрямую).
Нужно: **все PDF в проекте проходят через [markitdown](https://github.com/microsoft/markitdown)**
(Microsoft, конвертер office/pdf → Markdown). Markdown-выход стабильнее для выжимок:
сохраняет заголовки/списки/таблицы, не зависит от случайной разбивки строк в PDF.

Это часть конвейера «разбери inbox» (см. `бизнес/как-добавлять-материалы.md`):
PDF → markitdown → Markdown-текст → раскладка по «4 вёдрам» → выжимка в
`бизнес/материалы/обработанное/`.

## Исследование (что уже есть)

- Конвейер материалов описан в `бизнес/как-добавлять-материалы.md`. Для PDF там сказано
  просто «Claude прочитает файл» — конкретный инструмент не зафиксирован.
- `бизнес/inbox/**` в `.gitignore` — сырые PDF в git не едут, только выжимки. Значит
  утилита работает локально, её артефакт (`.md`) тоже не коммитится (если лежит в inbox).
- Конвенция утилит: `scripts/<имя>.py` с импортируемой логикой + `scripts/tests/test_<имя>.py`
  (пример — `scripts/list_strategies.py`). `scripts/tests` уже в `testpaths` pyproject.
- Тяжёлые/опциональные зависимости выносятся в optional-группы (`dev`, `dashboard`) —
  торговый runtime остаётся лёгким (AGENTS.md «малые модули»).

## Адекватизация (решения по объёму)

- **Зависимость — в новую optional-группу `materials`**, не в основные deps. Торговому боту
  markitdown не нужен; ставится только для разбора inbox: `pip install -e ".[materials]"`.
  Спека `markitdown[pdf]` — extra `pdf` тянет парсер PDF (pdfminer).
- **Утилита `scripts/pdf_to_md.py`**: один файл или вся папка `inbox/pdf/` → Markdown
  (stdout или `.md` рядом). markitdown импортируется **лениво** — модуль импортируется и
  тестируется без установленной зависимости; при реальном запуске без неё — понятная
  ошибка с командой установки.
- **Тесты** мокают конвертер (фейковый объект с `.convert().text_content`) — не требуют ни
  markitdown, ни реального PDF. Покрывают: валидацию расширения/наличия файла, проброс
  текста, обход папки, запись/печать.
- **`CLAUDE.md`/`AGENTS.md` не трогаю** (правило: только с явного «да»). Если нужно
  зафиксировать markitdown в разделе «Стек» CLAUDE.md — отдельно, с diff и согласием.

## 10 причин провала (критика)

1. **markitdown не установлен при запуске** → ленивый импорт + RuntimeError с командой `pip install -e ".[materials]"`.
2. **mypy strict падает на отсутствии стабов markitdown** → override `ignore_missing_imports` в pyproject; `_make_markitdown()` возвращает `Any`, текст приводится к `str`.
3. **Тесты требуют сети/реального PDF** → конвертер инъектируется фейком, файлы создаются в `tmp_path`.
4. **Скан-PDF без текстового слоя** → markitdown вернёт пустой/мусорный текст. Не цель этого плана (OCR — отдельно); вернуть что есть, пустой результат не маскировать.
5. **Конвертер пишет `.md` в `inbox` и он утечёт в git** → `inbox/**` в `.gitignore`, артефакт игнорится. Выходной путь по умолчанию — рядом с PDF.
6. **Имена с пробелами/кириллицей** → работаем через `pathlib.Path`, без shell-склейки.
7. **Огромный PDF** → markitdown читает весь файл; это ok для разовой разборки inbox, не runtime.
8. **Молчаливая потеря данных при ошибке одного файла в batch** → ошибки на файл логируются, exit-code ≠ 0, остальные обрабатываются.
9. **Переусложнение** → никакого daemon/watcher; простой CLI, вызывается вручную при «разбери inbox».
10. **Дубль логики чтения** → единая точка `convert_pdf()`, на неё ссылается инструкция; впредь PDF читаем через неё.
