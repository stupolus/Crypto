## Детали фазы 0.B (для самой сессии, не для брифа)

### §1. Структура `adapters/bingx/`

```
adapters/bingx/
├── __init__.py
├── config.yaml              # все числа: URLs, timeouts, rate limits,
│                            # precision, ссылки на источник
├── config.py                # загрузка конфига, валидация через pydantic
├── client.py                # BingXClient — базовый HTTP-клиент с подписью
├── public.py                # публичные REST-методы (serverTime, contracts,
│                            # ticker, klines)
├── websocket.py             # WS-каркас: коннект, gzip, Ping/Pong, reconnect
├── models.py                # pydantic-модели ответов (Contract, Ticker, Kline)
├── exceptions.py            # кастомные исключения адаптера
└── tests/
    ├── __init__.py
    ├── test_client.py       # подпись, retry, парсинг ошибок
    ├── test_public.py       # каждый публичный метод
    ├── test_websocket.py    # gzip-разжатие, Ping/Pong, реконнект (моки)
    └── fixtures/            # JSON-фикстуры реальных ответов docs
```

### §2. HTTP-клиент (`client.py`)

- Базовый URL берётся из конфига (`https://open-api.bingx.com` для live,
  `https://open-api-vst.bingx.com` для VST).
- Подпись HMAC-SHA256 по схеме из `plans/01` §3.
- Retry с экспоненциальным бэкоффом на 5xx и `429 Too Many Requests`.
- Парсинг бизнес-ошибок BingX (`code != 0`) → специфичное исключение.
- Лимиты: 100 запросов / 10 сек на эндпоинт (см. `plans/01` §7 п.18). Использовать
  библиотеку `aiolimiter` или собственный токен-бакет.
- Async, на `httpx`. Не `requests` — нам нужна интеграция с WS-частью.

### §3. Публичные REST (`public.py`)

Реализовать четыре метода:
- `get_server_time()` — для синхронизации часов перед подписью.
- `get_contracts()` — список всех контрактов USDT-M perpetual с их precision.
- `get_ticker(symbol)` — текущая цена и 24h-статистика.
- `get_klines(symbol, interval, limit)` — свечи. Учесть: интервал в REST формате
  (`15m`), не WS-формате (`15min`); максимум `limit=1440`.

Каждый метод:
- Type hints на вход и pydantic-модель на выход.
- Docstring со ссылкой на эндпоинт docs.
- Тест с фикстурой реального ответа (взять из ручной проверки через `curl`).

### §4. WebSocket-каркас (`websocket.py`)

- `wss://open-api-swap.bingx.com/swap-market` (см. `бизнес/инструменты-bingx.md`
  §«WS-endpoint»).
- Все ответы приходят в gzip — разжимать перед парсингом.
- Сервер шлёт `Ping` каждые 5 сек, наш клиент должен ответить `Pong` в течение
  10 сек, иначе разрыв. Реализовать через background-task.
- Лимит 200 каналов на коннект, 60 коннектов на IP. Сейчас не упрёмся, но
  заложить в архитектуру (один коннект на N символов с переподпиской).
- Подписка только на `market.kline.BTC-USDT.1m` — для проверки работоспособности.
- Реконнект с backoff при разрыве.

### §5. Тесты

Целевое покрытие: 70%+ на новые модули (`adapters/bingx/*.py`).

- `pytest` + `pytest-asyncio`.
- Реальные HTTP-запросы — только в отдельной группе `integration`, не в CI по
  умолчанию. Запуск: `pytest -m integration` — руками, отдельно.
- Моки HTTP — через `respx` (для httpx) или `aioresponses`.
