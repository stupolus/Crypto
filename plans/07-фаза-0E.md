# План 07 — Фаза 0.E: hardening (persistence + reconcile + security + метрики)

**Дата:** 2026-05-11
**Статус:** актуальный — план + реализация в той же сессии
**Связано:** [[plans/01-bingx-адаптер]] §11 фаза 0.E/§13, [[plans/06-фаза-0D-part2]], [[бизнес/риск-профиль]]

---

## 1. Контекст

Фаза 0.D part 2 закрыта: ордера + kill switch + user-stream + compensating-close работают. Не хватает:

- **Persistence:** при рестарте процесса ack-кэш в памяти теряется. Если рестарт между `place_order` и push-событием — стратегия может задублить ордер. Это блокер для live.
- **Reconcile:** на WS-разрыве события теряются. После reconnect нет синтетических sync-событий — стратегия не знает, что в этот момент произошло.
- **Логи небезопасны:** signed-запросы пока никто не логирует, но если добавить (для latency-метрик) — `signature` и `X-BX-APIKEY` уйдут в файл. Маскирование — обязательно до production.
- **Нет baseline-метрик:** не измерили реальные `latency place_order → ORDER_TRADE_UPDATE` и `slippage avgPrice vs last_price` на VST. Без них нет ориентира на live.

## 2. Цель плана

Закрыть **технический** hardening адаптера. После этого PR — переход к **плану первой стратегии** (BTC breakout).

1. **`OrderJournal`** (SQLite) — пишем каждое отправленное и подтверждённое состояние ордера. `place_order` → запись `pending`. Ack → `acked`. Push-событие → `filled`/`canceled`. При старте процесса — восстанавливаем кэш в памяти.
2. **Reconcile в `BingXUserDataStream`** — при успешном reconnect эмитим `UserStreamReconcileEvent` с актуальным `balance` / `positions` / `open_orders` (получены через REST). Стратегия должна интерпретировать как «состояние сейчас», не как «свежее событие».
3. **Маскирование секретов** — добавить `mask_signed_url`/`mask_headers` helper в `client.py`. Использовать в логах. Логирование самого `place_order` (latency) добавим, но с маской.
4. **Latency / slippage metrics** — на каждый `place_order(MARKET)` адаптер считает:
   - `latency_ms = update_time_ms - request_started_ms` (ack-latency).
   - `slippage_bps = (avg_price - request_mark_price) / mark_price * 10000` (для BUY: положительный = хуже; для SELL — наоборот).
   Запись в `ops/metrics.jsonl` (JSON-lines, простой формат).
5. **Security review checklist** — пройти `plans/01` §9.2 и зафиксировать статус.

**Что НЕ делаем в этой фазе:**
- ❌ Telegram-алерты — отдельный план (нужен бот, токен).
- ❌ Standalone stop_market/tp_market — после фазы 1 (если понадобится).
- ❌ Продвинутые метрики (heat-map, dashboards) — потом.
- ❌ Persistence positions/balance — это уже идёт через push-стрим, дублировать в SQLite не нужно.

## 3. Спецификация

### 3.1 `adapters/bingx/journal.py` — `OrderJournal`

Стек: SQLite через `sqlite3` (стандартная библиотека, без зависимостей). Чтобы не блокировать event-loop — синхронные вызовы заворачиваем в `asyncio.to_thread`.

Схема (одна таблица `orders`):
```
CREATE TABLE IF NOT EXISTS orders (
  client_order_id   TEXT PRIMARY KEY,
  exchange_order_id TEXT,
  symbol            TEXT NOT NULL,
  side              TEXT NOT NULL,
  type              TEXT NOT NULL,
  status            TEXT NOT NULL,   -- pending / acked / filled / partially_filled / canceled / rejected / failed
  quantity          TEXT NOT NULL,   -- Decimal stringified
  price             TEXT,
  attached_sl       TEXT,
  attached_tp       TEXT,
  ack_payload       TEXT,            -- JSON dump ack
  created_at_ms     INTEGER NOT NULL,
  updated_at_ms     INTEGER NOT NULL,
  last_event_type   TEXT,
  failure_reason    TEXT
);
CREATE INDEX IF NOT EXISTS idx_orders_symbol ON orders(symbol);
CREATE INDEX IF NOT EXISTS idx_orders_status ON orders(status);
```

API:
```
class OrderJournal:
    def __init__(self, db_path: Path): ...

    async def record_pending(self, req: OrderRequest, client_order_id: str) -> None: ...
    async def record_ack(self, client_order_id: str, ack: OrderAck) -> None: ...
    async def record_failure(self, client_order_id: str, reason: str) -> None: ...
    async def update_from_event(self, event: OrderUpdateEvent) -> None: ...
    async def get(self, client_order_id: str) -> JournalEntry | None: ...
    async def list_pending(self, symbol: str | None = None) -> list[JournalEntry]: ...
```

Интегрируется в `PrivateAPI.place_order`:
1. `record_pending` до отправки.
2. После ack — `record_ack`. На исключение — `record_failure`.
3. Стратегия / risk-engine при старте читает `list_pending` чтобы понять, есть ли неподтверждённые отправки.

### 3.2 Reconcile в `BingXUserDataStream`

Добавляем модель:
```
class UserStreamReconcileEvent(_StrictModel):
    event_type: Literal["RECONCILE"]
    event_time_ms: int
    balances: list[Balance]
    positions: list[Position]
    open_orders: list[Order]
```

`UserStreamEvent = OrderUpdateEvent | AccountUpdateEvent | UserStreamReconcileEvent`.

Поведение `BingXUserDataStream`:
- При успешном WS-подключении (после `_open_listen_key + _run_session start`) **сразу** делаем reconcile: вызываем `get_balance + get_positions + get_open_orders`, формируем `UserStreamReconcileEvent`, пушим в queue **перед** обычными WS-событиями.
- Стратегия видит первое событие как RECONCILE и инициализирует/синкает своё состояние.

Это даёт: после reconnect — снова RECONCILE с актуальным состоянием.

### 3.3 Маскирование секретов

Helper в `client.py`:
```
_MASK = "***"

def _mask_signed_query(query: str) -> str:
    """Заменяем значения signature и любых API-ключей на ***."""

def _mask_api_key_header(headers: Mapping) -> Mapping:
    """X-BX-APIKEY → *** в логах."""
```

Использование: на каждый успешный signed-запрос — `logger.debug` с маскированными URL и headers. По умолчанию уровень DEBUG, в проде включается явно.

### 3.4 Latency / slippage метрики

`adapters/bingx/metrics.py` — простой writer JSON-lines.

```
@dataclass
class OrderMetric:
    client_order_id: str
    symbol: str
    side: OrderSide
    type: EntryOrderType
    request_started_ms: int
    request_mark_price: Decimal | None      # для slippage; None если не задан
    ack_received_ms: int
    ack_avg_price: Decimal | None
    ack_status: str
    latency_ms: int
    slippage_bps: Decimal | None
```

В `place_order` — захватываем `request_started_ms` до `request_signed`, после ack считаем дельту. `request_mark_price` опциональный (стратегия передаст), без него slippage не считаем. Метрика пишется в `ops/metrics.jsonl` (асинхронно через `asyncio.to_thread`).

`ops/metrics.jsonl` уже игнорится через .gitignore (data/* как фактические данные). Baseline-summary в `ops/baseline-metrics.md` — будет заполняться эмпирически из накопленного.

## 5. Резюме

**Было:** адаптер технически работает на VST, но не годится для live: рестарт = потеря ack-кэша; WS-разрыв = потеря событий; логи могут утечь signature/key; нет baseline-метрик.

**Будет:** persistence ордеров в SQLite, reconcile-событие на каждом reconnect, маскирование секретов, базовые метрики latency/slippage. Адаптер готов к фазе 1 (BTC breakout 4 недели demo).
