# План 22 — Мульти-скрин активов и источники данных

## Проблема (из диалога 2026-05-15)

Пользователь справедливо спросил: почему бот не скринит много монет на
фьючах, не подключает доп. индикаторы Coinglass, TG-скринеры, новости,
DEX, Hyperliquid. Ответ из дисциплины проекта:

1. **Чужой сигнал ≠ триггер** (CLAUDE.md принцип №1). TG-скринеры,
   новости, DEX идут путём: парсим → копим статистику → проверяем
   edge → встраиваем как фильтр/фичу. Никогда как вход.
2. **Узкое место — не число монет, а edge + глубина выборки.**
   Скрин 2026-05-15 (7 монет, 2 ценовые стратегии): robust edge нет.
   `btc_breakout`/15m прибылен только на BTC (OOS PF 1.74, но 21
   сделка < порога 30) и случайно XRP — оверфит/шум. `trend_ema_4h`
   убыточен везде, 3–20 сделок (4h+360д → невалидно).
   Глубина данных: 15m ~5.5 мес, 4h ~360 д — мало для значимости.

Критерий приёмки edge (план 20): PF > 1.3 И Sharpe > 0.8 И ≥30 сделок
OOS, без подгонки параметров.

## Фазы

### 22.A — Глубина и ширина юниверса (бэктест, нулевой риск)
- Догрузить историю до 24 мес по топ-ликвидным перпам BingX:
  BTC ETH SOL BNB XRP DOGE ADA AVAX LINK LTC TRX DOT SUI TON.
  `scripts/download_klines.py` уже это умеет (`--months`).
- Интервалы: 15m (значимая выборка) + 4h (свинг/Щукин).
- Прогон всех ценовых стратегий через `run_batch_backtest` →
  сравнительная таблица. Цель: найти где вообще есть проблеск edge.

### 22.B — Полная поверхность Coinglass HOBBYIST
- Аудит: какие ещё эндпойнты доступны на $29 кроме
  liquidation/OI/CVD/funding (long/short ratio, top-trader
  positions, aggregated funding). Подключить доступное как
  фичи/фильтры в `parsers/coinglass/client.py`.
- Документировать в client.py docstring (как уже сделано для
  подтверждённых живым ключом эндпойнтов).

### 22.C — Сборщики данных (НЕ триггеры) в edge-pipeline
- **Hyperliquid** (в стеке как источник RWA ~52% рынка): коллектор
  OI/funding/цен по RWA-перпам.
- **DEX**: объёмы/листинги как фича аномалий.
- **Новости/макро**: уже есть `parsers/macro` — расширить охват.
- **TG-скринеры** (показанные в TG): парсеры → накопление →
  бэктест edge как фильтра. Только после статистики — в стратегию.
- Каждый источник: parse → store → backtest edge → решение
  «фильтр/фича/отбросить». Решение фиксируется в этом файле.

### 22.D — Зависимости от пользователя
- Материалы Щукина (присылает) → обновить выжимки, формализацию.
- Апгрейд тарифа Coinglass → полный 2-летний IS/OOS
  `liquidation_reversal`.

## Принципы (жёстко)
- Никакой подгонки под бэктест. Параметр обоснован логикой.
- Чужой сигнал — фильтр/фича после статистики, не триггер.
- Каждая фаза = коммит. Закрытая тема = PR.
- Деплой в Манус — только после подтверждённого edge, явным флагом.
