- Стратегия ЗАФИКСИРОВАНА: Faber 200SMA по индексу (Yahoo ^NDX),

- Идемпотентность: реконсиляция от ФАКТИЧЕСКОЙ позиции VST
  (не локальный флаг) → cron-safe, без двойных ордеров,
  самовосстановление.

  ≤ эквити, т.е. ≤1x — внутри лимитов CLAUDE.md ≤5x). Точная
  привязка к риск-профиль.md — follow-up 41.4 (Faber long-cash
  не имеет ценового стопа; стоп = флип сигнала, не цена).
- Критичный путь покрыт тестом (решение сигнал→действие +
  vst-guard), без сети.

## Исполнитель (41.2) — `scripts/faber_vst_executor.py`

1. BingXSettings → assert env==vst (иначе stop).
2. Faber-сигнал (LONG/CASH) с глубокого индекса.
3. `PrivateAPI`: get_balance (demo-эквити), get_positions(перп).
4. Target: LONG → нотионал = эквити·FRACTION / perp_price;
   CASH → flat.
5. Реконсиляция к target от факт. позиции:
   - уже соответствует → no-op (идемпотентно);
   - нужен flat → `close_position`;
   - нужен LONG из flat → `place_order` MARKET BUY qty;
   - смена → close, затем open.
6. Структурный лог `ops/faber_vst.jsonl` (дата, сигнал, эквити,
   target, действие, ack/err). `--dry` для инспекции без ордера.

## 10 причин провала (априорно, честно)

1. VST-ликвидность TradFi-перпа тонкая → проскальзывание.
2. Перп 24/7 vs индекс с часами → сигнал/исполнение рассинхрон.
3. BingX VST-квирки ордеров (видно в test_int_orders) —
  переиспользуем протестированный путь, не свой.
4. Faber-сделки редки → за demo может не быть ни одной смены.
5. Идемпотентность критична: баг = спам demo-ордеров (на
  demo не потеря денег, но невалидный demo) → тест обязателен.
6. Сайзинг-доля груба до 41.4 — на demo приемлемо, на live нет.
7. Yahoo-индекс задержка/ревизия → шум сигнала.
8. Сетевые/auth-сбои VST → ловим, лог, no-op (не падать).
9. demo ≠ live латентность/слиппедж — demo необходимо, не
  достаточно (как 40).
10. Даже чистый demo = право на разговор о live, НЕ live.

## 41.6 (план) — пред-входная оценка liquidation_price

Цель: закрыть последний пред-live риск-нюанс из 41.5 —
активировать liq-buffer RiskEngine ДО входа, передав
консервативную оценку `liquidation_price` в RiskInputs.

Подход (изолированная маржа, linear USDT-перп, BingX):
- LONG: `liq ≈ entry · (1 − 1/L + mmr)`, где L — эффективное
  плечо, mmr — ставка поддерживающей маржи.
- Консервативность (буфер-чек должен быть строгим, не мягким):
  берём L = потолок плеча (B-tier ≤3x — максимум, что движок
  вообще пропустит; фактическое risk-based плечо ниже → реально
  liq дальше → оценка пессимистична и безопасна). mmr — берём
  консервативно высокую ставку из конфига.
- Chicken-egg (liq нужен ВХОДОМ в evaluate, а точное плечо —
  ВЫХОД) решается тем, что оценка не зависит от approval:
  считается от потолка плеча, а не от факт. сайзинга.

Числа — НЕ в коде (AGENTS.md): `maintenance_margin_rate`
добавляется в `бизнес/риск-профиль.md` (источник истины риск-
цифр) и в `core/risk/config.yaml`/LimitsConfig. Источник —
публичная спецификация BingX maintenance-margin для мажоров,
взятая с запасом (выше реальной tier-1 ставки).

Артефакты:
- `core/risk/config.py`: поле `maintenance_margin_rate` в
  LimitsConfig (Field gt=0, lt=1).
- `core/risk/config.yaml`: значение + ссылка на риск-профиль.
- `бизнес/риск-профиль.md`: строка про mmr и формулу liq.
- `scripts/faber_vst_executor.py`: чистая функция
  `estimate_liq_price(entry, side, lev_cap, mmr)`; результат →
  `RiskInputs.liquidation_price`. Логируем оценку в jsonl.
- `scripts/tests/test_faber_vst_decide.py`: тесты на формулу
  (LONG, монотонность по L и mmr, граница буфера).

Стопы: только VST (env-guard прежний). Не меняет торговую
логику Faber — только активирует уже существующий
liq-buffer гейт раньше (строже, безопаснее). Live по-прежнему
требует отдельного явного «да» + demo-критерий (план 40).
