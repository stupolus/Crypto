## Жёсткая рамка (безопасность, defense-in-depth)

- ТОЛЬКО VST. В коде HARD-assert `settings.env == "vst"` ДО
  любого ордера — отказ, если не vst (поверх конфиг-изоляции).
- Live ЗАПРЕЩЁН. Переход на live — отдельное явное «да» ПОСЛЕ
  пройденного demo по зафикс. критерию плана 40 (CLAUDE.md).
- Стратегия ЗАФИКСИРОВАНА: Faber 200SMA по индексу (Yahoo ^NDX),
  long-or-cash. Сигнал по вчерашнему закрытию (без look-ahead).
- Идемпотентность: реконсиляция от ФАКТИЧЕСКОЙ позиции VST
  (не локальный флаг) → cron-safe, без двойных ордеров,
  самовосстановление.
- Сайзинг: консервативная фикс-доля demo-эквити (нотионал
  ≤ эквити, т.е. ≤1x — внутри лимитов CLAUDE.md ≤5x). Точная
  привязка к риск-профиль.md — follow-up 41.4 (Faber long-cash
  не имеет ценового стопа; стоп = флип сигнала, не цена).
- Критичный путь покрыт тестом (решение сигнал→действие +
  vst-guard), без сети.

## Исполнитель (41.2) — `scripts/faber_vst_executor.py`

1. BingXSettings → assert env==vst (иначе stop).
2. Faber-сигнал (LONG/CASH) с глубокого индекса.
3. `PrivateAPI`: get_balance (demo-эквити), get_positions(перп).
4. Target: LONG → нотионал = эквити·FRACTION / perp_price;
   CASH → flat.
5. Реконсиляция к target от факт. позиции:
   - уже соответствует → no-op (идемпотентно);
   - нужен flat → `close_position`;
   - нужен LONG из flat → `place_order` MARKET BUY qty;
   - смена → close, затем open.
6. Структурный лог `ops/faber_vst.jsonl` (дата, сигнал, эквити,
   target, действие, ack/err). `--dry` для инспекции без ордера.

## 10 причин провала (априорно, честно)

1. VST-ликвидность TradFi-перпа тонкая → проскальзывание.
2. Перп 24/7 vs индекс с часами → сигнал/исполнение рассинхрон.
3. BingX VST-квирки ордеров (видно в test_int_orders) —
  переиспользуем протестированный путь, не свой.
4. Faber-сделки редки → за demo может не быть ни одной смены.
5. Идемпотентность критична: баг = спам demo-ордеров (на
  demo не потеря денег, но невалидный demo) → тест обязателен.
6. Сайзинг-доля груба до 41.4 — на demo приемлемо, на live нет.
7. Yahoo-индекс задержка/ревизия → шум сигнала.
8. Сетевые/auth-сбои VST → ловим, лог, no-op (не падать).
9. demo ≠ live латентность/слиппедж — demo необходимо, не
  достаточно (как 40).
10. Даже чистый demo = право на разговор о live, НЕ live.

## 41.3 ВЫПОЛНЕНО (2026-05-18) — позиция открыта на VST demo

Цикл «гоняй→ошибка→фикс→снова» (санкционировано, VST/demo):
1. OrderRequest отклонил «нет стопа» (риск-профиль.md) → стоп
   = уровень SMA200 (собственный выход Faber) + сайзинг ОТ
   РИСКА 1% эквити (Принцип 2), плечо ≤5x.
2. APIError 109400 one-way → positionSide=BOTH.
3. APIError 101414 «non-crypto one-way не поддержан openapi» →
   set_position_mode(hedge) + positionSide=LONG.
4. ✅ Ордер прошёл: order_ack 20564676..., позиция
   **LONG 0.26 @ 28999**, стоп STOP_MARKET @ 25108 (=SMA200)
   на бирже, риск 1% эквити.
5. Идемпотентность подтверждена: повтор → `noop` (cron-safe).
6. Баг адаптера: `Order.orderId` (get_open_orders) без int→str
   коэрции — пойман на demo, ПОЧИНЕН (private_models.py),
   111 тестов адаптера зелёные.

Статус: Faber реально исполняется на BingX VST demo-депозите.
Раннер идемпотентен и реконсилирует ежедневно (CASH→close,
LONG→open снова — «открывай снова» встроено). ЭТО НЕ LIVE.
Вердикт по критерию плана 40 — через ≥4 нед (план 41.4).
Ошибки каждого прогона пишутся в ops/faber_vst.jsonl
(механизм «записывай ошибки» для текущего мониторинга).

## 41.4 (2026-05-18) — усиление автономного demo-цикла

- Сайзинг сверен с бизнес/риск-профиль.md (источник истины,

считает сайзинг инлайн, НЕ через core RiskEngine — поэтому

demo для этого). ОБЯЗАТЕЛЬНО до любого разговора о live:
полная интеграция RiskEngine (liq-buffer + дневной −3% /
недельный −7% брейкеры + макс 3 сделки/день). Это —
предусловие live, не demo.
