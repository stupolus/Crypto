# План 05 — Фаза 0.D (part 1): ордера и kill switch

**Дата:** 2026-05-11
**Статус:** актуальный — план + реализация в той же сессии
**Связано:** [[plans/01-bingx-адаптер]] §4.1/§4.2/§7 п.6–7/§11 фаза 0.D, [[plans/04-фаза-0C]], [[бизнес/риск-профиль]]

---

## 1. Контекст

0.C закрыта: подпись HMAC-SHA256 верифицирована на VST, account готов (ISOLATED, one-way, leverage 3x для BTC-USDT), баланс пополнен на 100 000 VST. Все приватные read-методы и idempotent-setters работают.

Не сделано: ни одного write-метода (отправка ордера, отмена, закрытие позиции). Stream user-events и `cancelAllAfter` отложены на 0.D part 2.

## 2. Цель плана

Реализовать минимально-достаточный набор write-методов с **жёстким инвариантом** «нет позиции без подтверждённого SL на бирже»:

1. `place_order` (market, limit) — entry с **обязательным attached_stop_loss**, либо `reduce_only` для закрытия.
2. `cancel_order(symbol, order_id)` / `cancel_all(symbol)`.
3. `close_position(symbol)` — kill switch: snimaет защитные ордера + reduce_only market в обратную сторону.
4. Integration на VST: `open(market) → assert position → cancel attached SL → close → assert flat → assert fill записан`.

**Что НЕ делаем сейчас (отложено на 0.D part 2):**
- ❌ `stream_user_events` (User Data Stream, listenKey).
- ❌ `Cancel All After` (биржевой dead-man timer).
- ❌ Compensating-close fallback (если ack без SL — пока бросаем `OrderRejected`, фоновую компенсацию добавим вместе с user-data stream).
- ❌ stop_market / stop_limit как отдельные типы (через attached SL/TP на entry — уже покрыто).
- ❌ persistence ack-кэша в SQLite (in-memory достаточно для smoke).

## 3. Спецификация

### 3.1 `OrderRequest` (новая модель в `private_models.py`)

Поля строго следуют plans/01 §4.1. Все Decimal-поля валидируются под `quantity_precision` / `price_precision` контракта вызывающей стороной — адаптер локально не округляет (риск разъезда с журналом).

```
class OrderRequest(_StrictModel):
    symbol: str                      # "BTC-USDT" (hyphen-валидатор)
    side: Literal["BUY", "SELL"]     # "BUY" = открытие LONG / закрытие SHORT
    position_side: Literal["LONG", "SHORT"]  # one-way: всегда LONG для BUY, SHORT для SELL
    order_type: Literal["MARKET", "LIMIT"]
    quantity: Decimal
    price: Decimal | None = None     # обязателен для LIMIT
    reduce_only: bool = False
    time_in_force: Literal["GTC", "IOC", "FOK"] = "GTC"
    attached_stop_loss: Decimal | None = None    # NEW позиция: обязателен (если не reduce_only)
    attached_take_profit: Decimal | None = None  # NEW позиция: опционально
    client_order_id: str | None = None  # 1-40 chars, lowercase BingX-side
```

**Инвариант (проверяется в `place_order`):**
- `reduce_only=False` + нет `attached_stop_loss` → `ValueError` («open position without attached SL запрещён»).
- `attached_stop_loss` при `reduce_only=True` → `ValueError` («close-side не несёт защитные ордера»).
- `order_type="LIMIT"` без `price` → `ValueError`.
- `quantity <= 0` → `ValueError`.

### 3.2 `PrivateAPI.place_order(req) -> Order`

POST `/openApi/swap/v2/trade/order`. Параметры в alpha-sorted порядке (квирк §7 п.31). `recvWindow` НЕ отправляем (квирк §7 п.30).

**Атомарность entry+SL:** `stopLoss` (и `takeProfit` если задан) сериализуются как stringified JSON (квирк §7 п.7):
```
stopLoss='{"type":"STOP_MARKET","stopPrice":62100,"workingType":"MARK_PRICE"}'
```
`workingType: MARK_PRICE` фиксирован (защита от stop-hunt на спот-цене, см. plans/01 §8).

Возвращает `Order` (модель из 0.C). Поля BingX `data.order` достаются и парсятся.

### 3.3 `cancel_order(symbol, order_id) -> Order`

DELETE `/openApi/swap/v2/trade/order` с `orderId=...&symbol=...`. Возвращает обновлённый `Order` (статус `CANCELED`). Если ордер уже исполнился — BingX отдаёт `code=80018`/`80019` (уточняется на VST). Маппим в `OrderNotFound` / `OrderAlreadyFilled` (новое подсемейство `BingXError`).

### 3.4 `cancel_all(symbol) -> list[Order]`

DELETE `/openApi/swap/v2/trade/allOpenOrders` с `symbol=...`. Возвращает список отменённых.

### 3.5 `close_position(symbol) -> Order | None`

1. `await cancel_all(symbol)` — снимаем все защитные ордера.
2. `pos = await get_positions(symbol)` — узнаём direction и size.
3. Если `pos.position_amt == 0` → `None` (нечего закрывать, идемпотентно).
4. Иначе `place_order` с `side` противоположный позиции, `reduce_only=True`, `order_type="MARKET"`, `quantity=abs(position_amt)`, без attached SL.

### 3.6 Эндпоинты в `config.yaml`

Добавляем в `rest_endpoints`:
- `place_order: /openApi/swap/v2/trade/order`
- `cancel_order: /openApi/swap/v2/trade/order` (DELETE на тот же путь)
- `cancel_all_orders: /openApi/swap/v2/trade/allOpenOrders`

## 4. Защита

- Все write-вызовы — только на `env=vst`. В `place_order` ассерт: `self._client.config.env == "vst"` для этой фазы. Снимется явным «разрешено live» в коммите перед фазой 1.
- `client_order_id` — если не передан, генерим `uuid4().hex[:32]`. Идемпотентность.
- Integration-тест очищает за собой: в `finally` зовёт `close_position` + `cancel_all`, даже если упал.

## 6. Резюме

**Было:** ноль write-методов в адаптере, инвариант «нет позиции без SL» — только в декларациях, не в коде.

**Будет:** `place_order` отправляет entry + SL атомарным POST, в одном теле; `close_position` снимает защитки и закрывает позицию; на VST полный цикл проходит без падений.

**Альтернатива «сразу сделать всё, включая user-data stream»:** нет — это удваивает объём. Кусок stream-а и dead-man-timer — естественно отделяемая часть, и она важна для **долгоживущего** бота, а не для smoke. Сначала proof-of-life на orders, потом стабильность.
