# План 03: Obsidian-vault поверх `бизнес/`

**Дата:** 2026-05-10
**Статус:** актуальный — план + одна фаза реализации в той же сессии
**Автор:** Claude + пользователь
**Связано:** [[plans/00-стратегия-проекта-2026-05-09]], [[бизнес/INDEX]], [[бизнес/материалы/каналы/dmitry-shukin-crypto]] (там описан принцип «4 ведра» для выжимки текстов)

---

## 1. Контекст

Папка `бизнес/` уже работает как «второй мозг»: markdown-файлы, wiki-links `[[имя]]`, индекс в `бизнес/INDEX.md`. Это совместимо с Obsidian, но не оформлено как полноценный vault и не покрывает рабочий процесс с **сырыми материалами**.

Проблема, которую решаем: пользователь регулярно получает материалы по трейдингу разных типов — PDF (книги, отчёты), видео (лекции, ролики), скриншоты (графики, экраны бирж, скрины TG), голосовые записи (свои мысли), текстовые экспорты из Telegram, ссылки на статьи. Сейчас:

- Нет понятного **места**, куда их класть.
- Нет **правил** — куда какой тип файла, как назвать.
- Нет **пайплайна обработки** — что Claude делает с каждым типом.
- Нет **шаблонов** Obsidian для типовых сущностей (видео-выжимка, статья-выжимка, торговая идея).
- Сырые большие файлы рискуют попасть в git, если кто-то скажет `git add -A`.

Без этого материалы будут оседать в Downloads / Telegram и не превращаться в знание. Стратегия проекта (CLAUDE.md, принцип №5: «каждое решение — в файлы») без работающего vault'а ломается на стыке «у пользователя в голове идея → файл в репо».

## 2. Цель плана

Запустить `бизнес/` как **полноценный Obsidian-vault с inbox-конвейером**:

1. Пользователь скидывает любой материал в одну из подпапок `бизнес/inbox/<тип>/` — никакой ручной разметки.
2. Claude в новой сессии читает inbox, обрабатывает материалы по типам (Whisper для аудио/видео, OCR для скринов, выжимка по «4 ведрам» для текстов — как в [[бизнес/материалы/каналы/dmitry-shukin-crypto]]).
3. Готовые выжимки переезжают в `бизнес/материалы/обработанное/<категория>/<название>.md` со ссылками `[[имя]]` на основные правила/риск-профиль/идеи.
4. Сырые большие файлы остаются локально (gitignore), в репо коммитятся только итоговые markdown-выжимки.
5. Vault открывается в Obsidian без ручной настройки (минимально валидный `.obsidian/`).

**Что НЕ делаем в этом плане:**

- ❌ Не пишем код пайплайна обработки (Whisper, OCR — это будет отдельный план, когда понадобится автоматизация). Сейчас обработка — ручная Claude-сессия по чек-листу.
- ❌ Не настраиваем синхронизацию vault'а через Obsidian Sync / iCloud — пользователь работает в репо локально + git push.
- ❌ Не лезем в код адаптера BingX (это [[plans/02-фаза-0B]]).
- ❌ Не плодим плагины Obsidian (community plugins — отдельная история, в gitignore уже есть).

### NICE TO HAVE (не сейчас)

- Автоматический скрипт обработки inbox (Whisper API + OCR + распознавание типа файла) — отдельный план, после того как накопится 5–10 реальных кейсов и будет понятен формат.
- Dataview-плагин для автогенерации списков обработанных материалов (community plugin, в репо не коммитим — пользователь ставит в Obsidian локально).
- Шаблон ретроспективы сессии (это уже есть в `retro/`, дублировать не надо).

### Чего НЕ должно быть в vault'е

- ❌ Сырых видео/аудио/PDF >10MB в git (только локально через inbox).
- ❌ API-ключей, токенов, скринов с открытыми ключами/email/балансами без редакта (это правило в инструкцию).
- ❌ Дублирования цифр риск-профиля в шаблонах — только wiki-link `[[риск-профиль]]`.

### Структура папок после выполнения плана

```
│   ├── app.json
│   ├── appearance.json
│   ├── core-plugins.json
│   ├── community-plugins.json
│   └── templates.json
├── бизнес/                         ← корень vault'а
│   ├── INDEX.md                    ← обновлён
│   ├── как-добавлять-материалы.md  ← НОВЫЙ — инструкция
│   ├── цели.md
│   ├── риск-профиль.md
│   ├── идеи.md
│   ├── инструменты-bingx.md
│   ├── правила-торговли/...
│   ├── inbox/                      ← НОВОЕ — сюда сырые файлы (gitignored)
│   │   ├── pdf/.gitkeep
│   │   ├── video/.gitkeep
│   │   ├── audio/.gitkeep
│   │   ├── screenshots/.gitkeep
│   │   ├── text/.gitkeep
│   │   └── links/.gitkeep
│   ├── templates/                  ← НОВОЕ — шаблоны Obsidian (коммитим)
│   │   ├── шаблон-видео-выжимка.md
│   │   ├── шаблон-статья-выжимка.md
│   │   ├── шаблон-торговая-идея.md
│   │   └── шаблон-заметка-дня.md
│   └── материалы/
│       ├── каналы/...
│       ├── скринеры/...
│       └── обработанное/.gitkeep   ← НОВОЕ — сюда выжимки
```

### Почему так

- **`inbox/` внутри `бизнес/`, а не в корне репо.** Это часть «второго мозга», логически принадлежит ему. Если делать в корне — придётся лазать по двум местам.
- **Подпапки по типу файла, а не «всё в одну `inbox/`».** Тип файла → разный пайплайн обработки. Папка = первичная классификация без ручной разметки.
- **`материалы/обработанное/` отдельно от `материалы/каналы/`, `материалы/скринеры/`.** Существующие `каналы/` и `скринеры/` — это **описания источников** (какой канал, кто автор, статус). `обработанное/` — это **выжимки конкретных материалов**: один PDF = один файл, одно видео = один файл.
- **`templates/` в `бизнес/`, а не `.obsidian/templates/`.** Чтобы шаблоны коммитились в репо и были видны как обычные markdown-файлы, а не прятались в `.obsidian/`.
- **`.obsidian/` коммитим, но в `.gitignore` уже исключены `workspace*`, `cache`, `plugins`, `themes`** — личные настройки и тяжёлые плагины не утекают, общая структура vault'а сохраняется.

### Конвенция именования файлов в `обработанное/`

- `<категория>/<краткое-название-латиницей>-<YYYY-MM-DD>.md` — например, `обработанное/видео/shukin-vsa-basics-2026-05-12.md`.
- Категории: `видео/`, `статьи/`, `книги/`, `скрины/`, `аудио/`, `tg-экспорты/`.
- Кириллица в имени допустима, но латиница безопаснее для git/cli/CI. Решает Claude при обработке.

## 5. Пайплайн обработки по типам

Это спецификация для **Claude в будущей сессии**, не для автоматического скрипта. Когда пользователь говорит «разбери inbox» — Claude действует по этому пайплайну.

### 5.1. PDF (`inbox/pdf/`)

Источник: книги, отчёты, white papers.

1. Читаем PDF (Read tool с параметром `pages`, по 20 страниц за заход — лимит инструмента).
2. Раскладка по «4 ведрам» (см. [[бизнес/материалы/каналы/dmitry-shukin-crypto]] §«План обработки»):
   - ✅ Тестируемое правило → в выжимку как «правила входа/выхода/риска».
   - 🟡 Контекстная мудрость → в раздел «принципы».
   - ⚠️ Догма без доказательств → в раздел «что проверить».
   - ❌ Маркетинг/мотивация → не сохраняем.
3. Создаём `бизнес/материалы/обработанное/книги/<автор>-<кратко>-<дата>.md` по шаблону `шаблон-статья-выжимка.md`.
4. Если в книге есть конкретные пороги/числа, которых нет в `риск-профиль.md` — НЕ хардкодим в выжимку, а добавляем как «гипотезу к проверке» в раздел `что-проверить.md` в `обработанное/`.
5. Сырой PDF остаётся в `inbox/pdf/` — пользователь сам решает, удалить или оставить локально (он в gitignore, в репо не уйдёт). Опционально: переименовываем в `inbox/pdf/_processed_<имя>.pdf` чтобы видеть что уже разобрано.

### 5.2. Видео (`inbox/video/`)

Источник: YouTube-лекции, ролики из TG, записи вебинаров.

1. Транскрипция через Whisper (локальная модель `whisper-large-v3` или OpenAI API — решает пользователь, в плане не выбираем).
2. Транскрипт сохраняем во временный `.txt` рядом с видео (тоже в gitignore).
3. Выжимка по «4 ведрам», как PDF.
4. Создаём `бизнес/материалы/обработанное/видео/<автор>-<кратко>-<YYYY-MM-DD>.md` по шаблону `шаблон-видео-выжимка.md`.
5. В выжимке обязательно: длительность, источник (URL или «от пользователя»), таймкоды ключевых моментов.
6. Связываем с описанием канала (если есть): wiki-link на `[[материалы/каналы/<канал>]]`. Если канала ещё нет — создаём минимальное описание там же.

### 5.3. Аудио (`inbox/audio/`)

Источник: голосовые из Telegram, диктофон пользователя, подкасты.

1. Если это голосовая заметка пользователя (мысли вслух) — Whisper → текст → разбираем как «торговую идею» (шаблон `шаблон-торговая-идея.md`) или «заметку дня» (шаблон `шаблон-заметка-дня.md`).
2. Если это подкаст/чужая запись — как видео (выжимка по 4 ведрам).
3. Файл назначения: `бизнес/материалы/обработанное/аудио/<кратко>-<дата>.md` или `бизнес/идеи.md` (для коротких мыслей пользователя — добавляем в существующий backlog).

### 5.4. Скриншоты (`inbox/screenshots/`)

Источник: скрины графиков, торговых экранов, TG-сообщений.

1. Read tool читает изображения нативно — пользуемся.
2. Если это **скрин графика** (свечи, индикаторы):
   - Описываем что на нём: инструмент, таймфрейм, паттерн, кластер цен.
   - Если пользователь добавил вопрос — отвечаем рядом.
   - Сохраняем как `обработанное/скрины/<инструмент>-<паттерн>-<дата>.md`. Сам PNG оставляем локально или копируем в `обработанное/скрины/_assets/` (но это уже коммитится — следить за размером, < 500KB).
3. Если это **скрин TG-сообщения / новости / ордера**:
   - Транскрибируем текст, фиксируем источник (канал, дата если видна).
   - Если это сигнал из TG — НЕ копируем как «сделать», а добавляем в статистику источника (потом будет парсер).
4. **Безопасность:** перед коммитом скрина проверяем, нет ли на нём API-ключей, email, балансов в долларах, имени пользователя. Если есть — закрашиваем (или вообще не коммитим, оставляем только текстовую выжимку).

### 5.5. Текст (`inbox/text/`)

Источник: TG-экспорты (json/html), копипаст статей, заметки в .md/.txt.

1. TG-экспорт (json) — парсим, выделяем посты конкретных каналов, прогоняем через шаблон выжимки канала. Складываем в `обработанное/tg-экспорты/<канал>-<период>.md`.
2. Статья (txt/md/html) — выжимка по 4 ведрам, шаблон `шаблон-статья-выжимка.md`. Сохраняем в `обработанное/статьи/<источник>-<кратко>-<дата>.md`.
3. Просто заметки пользователя — короткие в `идеи.md` или `заметки-дня.md`, длинные в `обработанное/заметки/<дата>-<тема>.md`.

### 5.6. Ссылки (`inbox/links/`)

Источник: пользователь скидывает ссылки на статьи/видео/Twitter-треды.

1. Файл в этой папке — это просто `.md` или `.txt` со списком URL (по одному на строку), либо одиночная ссылка.
2. Claude использует WebFetch (для статей) или просит пользователя скачать видео в `inbox/video/`.
3. После обработки — выжимка идёт в соответствующую категорию `обработанное/`, ссылка вычёркивается из файла-списка.

## 6. Что коммитим vs не коммитим

| Что | В git? | Почему |
|---|---|---|
| `бизнес/inbox/**` (сырые файлы) | ❌ | Большие, личные, могут содержать секреты. Только локально. |
| `бизнес/inbox/*/.gitkeep` | ✅ | Чтобы папки существовали в свежем clone. |
| `бизнес/материалы/обработанное/**` | ✅ | Это и есть конечный продукт второго мозга. |
| `бизнес/templates/**` | ✅ | Шаблоны нужны всем сессиям. |
| `бизнес/как-добавлять-материалы.md` | ✅ | Инструкция должна быть в репо. |
| `.obsidian/app.json, appearance.json, core-plugins.json, community-plugins.json, templates.json` | ✅ | Минимальный конфиг vault'а — чтобы открывалось «из коробки». |
| `.obsidian/workspace*, cache, plugins/, themes/` | ❌ (уже в .gitignore) | Личные/тяжёлые. |

Жёсткое правило: **скрины/PDF/видео > 10MB в git не пускаем.** Если размер критичен — оставляем выжимку и ссылку, а не файл.

## 7. MM-устойчивость

Не применимо. Это инфраструктура второго мозга, не торговая стратегия. MM-контр-стратегии (см. [[бизнес/правила-торговли/мм-контр-стратегии]]) — обязательная секция только в планах **торговых стратегий**.

## 8. 10 причин провала

1. **Пользователь кидает файлы, но не запускает «разбери inbox».** Inbox раздувается, ничего не обрабатывается.
   - Митигация: в `как-добавлять-материалы.md` — отдельный раздел «как сказать Claude разобрать». Простая фраза-триггер: «разбери inbox».

2. **Большой видеофайл случайно попадёт в git через `git add -A`.**
   - Митигация: в CLAUDE.md уже жёсткий запрет `git add -A` / `git add .` (раздел Git-процесс). `.gitignore` на `бизнес/inbox/**` — вторая линия защиты. Третья: визуальная проверка `git status` перед коммитом.

3. **Скрин с открытым API-ключом / email / балансом утечёт в публичный репо.**
   - Митигация: явное правило в `как-добавлять-материалы.md` — перед коммитом скринов всегда проверять. Claude в пайплайне 5.4 п.4 — обязательная проверка.

4. **Outdated шаблоны.** Шаблон `шаблон-видео-выжимка.md` через полгода не отражает реальный формат, который сложился в выжимках.
   - Митигация: раз в квартал ревизия шаблонов на основе того, что реально пишется. Точка контроля — добавить в чек-лист квартального retro.

5. **Wiki-links ломаются** при переименовании/перемещении файлов.
   - Митигация: Obsidian умеет авторефакторить ссылки при переименовании внутри vault'а, но только если правка через Obsidian. Claude правит через Edit/Write — ссылки ломаются молча. Поэтому: при переименовании файла — `grep -r "[[старое-имя]]"` и обновить все ссылки в той же сессии. В инструкции пользователю это **не выносим** (он не должен думать про это), но Claude должен помнить — добавлю в `как-добавлять-материалы.md` отдельный раздел «для Claude».

6. **Дублирование между `материалы/обработанное/` и существующими `материалы/каналы/`.**
   - Митигация: чёткое разделение, описано в §4 «Почему так». `каналы/` = описание источника, `обработанное/` = выжимка конкретных материалов из источника. В шаблонах прописываем wiki-link на канал.

7. **Vault не открывается в Obsidian** — кривой `.obsidian/` конфиг.
   - Митигация: используем минимальные дефолтные значения, без community plugins. Если что-то сломается — Obsidian сгенерирует свой `.obsidian/` поверх.

8. **Слишком много шаблонов → пользователь путается.**
   - Митигация: на старте 4 шаблона. Добавляем новые только когда реально нужны (3+ кейса). Пишем в `как-добавлять-материалы.md`: «если не уверен какой шаблон — заметка-дня по умолчанию».

9. **Пользователь не понимает русский путь `бизнес/inbox/audio/`** (например, на VPS Linux с не-UTF локалью).
   - Митигация: проверять, что VPS поддерживает UTF-8 (`locale` команда). Кириллица в путях — общий стиль проекта (см. CLAUDE.md), не меняем.

10. **Inbox превращается в свалку**, материалы теряются среди старых.
    - Митигация: правило «обработанное → перенесли в `обработанное/`, исходник переименовали в `_processed_<имя>` или удалили». Стандарт: после обработки `inbox/` должен опустеть.

## 9. Фазы реализации

### Фаза 1 (единственная, в этой же сессии — 30–60 минут)

Создание структуры + шаблоны + инструкция + .obsidian + .gitignore patch + INDEX update. Это не код, а файлы и папки — безопасно делать без отдельной сессии.

**Чек-лист:**

## 12. Резюме

**Что было:** `бизнес/` — markdown-файлы с правилами, ссылками, риск-профилем. Совместимо с Obsidian, но без inbox-конвейера и без места для сырых материалов. Любой PDF/видео/скрин от пользователя оседал «нигде».

**Что стало:** полноценный vault с конвейером `inbox → обработка → обработанное`, шаблонами для типовых выжимок, инструкцией для пользователя простыми словами и защитой от случайного коммита больших/секретных файлов.

**Какую проблему решал:** ликвидация разрыва между «у пользователя есть материал» и «материал стал частью второго мозга». До этого плана единственный путь был через ручное копирование Claude'у в чат, что не масштабируется.
