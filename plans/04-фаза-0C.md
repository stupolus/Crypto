## 1. Контекст

Фаза 0.B закрыта 2026-05-10 (см. retro): есть `adapters/bingx/` (config, exceptions, models, client, public, websocket) + 39 unit + 4 integration на live public, покрытие 83%. Подпись HMAC-SHA256 уже реализована в `BingXClient.request_signed` как **заготовка** — без ключей бросает `AuthError`. Приватных вызовов в коде ещё нет, моделей `Balance/Position/Order/Fill` нет, `.env`-чтения нет, time-sync не реализован.

Фаза 0.C — это первый раз, когда мы реально стучимся в BingX **с ключами**. Без неё нельзя приступать к ордерам (0.D), нельзя бэктестить на реальном балансе, нельзя проверить, что инварианты `ISOLATED` + `one-way` действительно стоят на аккаунте. Все эндпоинты — на **VST (demo trading)**, не на live. Это ключевое: пока подпись не верифицирована на демо, любой live-вызов запрещён правилами безопасности CLAUDE.md.

## 2. Цель плана

Запустить приватный read + idempotent-setters на VST:

1. Чтение ключей из `.env` (никогда из YAML, никогда из CLI-аргументов в коде).
2. Time-sync с серверным временем BingX (offset + периодический resync).
3. Smoke на VST: `connect → set_margin_mode → set_leverage → get_balance → disconnect` без падений.
4. Приватные read-методы: `get_balance`, `get_positions`, `get_open_orders`, `get_fills` — типизированные модели на `Decimal`.
5. Idempotent-setters: `set_margin_mode(ISOLATED)`, `set_leverage(symbol, n)`, `set_position_mode(one_way)` — обязательные пререквизиты для ордеров 0.D.
6. Unit-тесты (respx) + integration на VST + ≥70% coverage.

**Что НЕ делаем в этой фазе:**

- ❌ `place_order`, `cancel_order`, `close_position` — это 0.D.
- ❌ User-data WS-стрим (баланс/позиции/ордера push) — это тоже 0.D.
- ❌ Live-окружение. Только VST (`env: vst` в `config.yaml`).
- ❌ Любые манипуляции с реальными деньгами / перевод средств — у ключей запрещён вывод (правило безопасности CLAUDE.md).
- ❌ Сложная reconcile-логика — это 0.E.

## 4. Архитектура

### Слои

```
PrivateAPI (private.py)
        │ методы: get_balance, get_positions, get_open_orders, get_fills,
        │         set_margin_mode, set_leverage, set_position_mode
        ▼
BingXClient.request_signed (client.py)
        │ + ServerTimeSyncer (time_sync.py) → timestamp = now_ms()
        │ + sign_query (уже есть) → HMAC-SHA256
        │ + token-bucket (уже есть) → rate limit
        │ + retry/backoff (уже есть) → 429/5xx
        │ + спец-case: 109400 → re-sync time + 1 ретрай
        ▼
httpx.AsyncClient → BingX VST (open-api-vst.bingx.com)
```

### Источник ключей

```
.env (gitignored)        →   pydantic-settings BingXSettings
BINGX_VST_API_KEY=...        env_prefix="BINGX_"
BINGX_VST_API_SECRET=...     active_key / active_secret по env
BINGX_ENV=vst                                │
                                             ▼
                                       BingXClient(settings=...)
                                       или BingXClient(api_key=..., api_secret=...)
                                       (для тестов)
```

DI-приоритет: явные `api_key`/`api_secret` в конструкторе > `settings.active_*` > `None` (`AuthError` при попытке `request_signed`).

### Конфиг

`config.yaml` уже содержит `signing.server_time_resync_interval_s: 300` и `signing.recv_window_ms: 5000` — берём оттуда. **Новых чисел не добавляем.** Если выяснится что для какого-то setter'а нужен другой rate limit — это число уже в `rate_limits` (`set_margin_type_per_sec`, `set_leverage_per_sec`, `set_position_mode_per_sec`), просто подключим.

### Маппинг VST-эндпоинтов (расширение `RestEndpoints` в `config.yaml` + `config.py`)

Добавляем в `rest_endpoints`:

- `balance: /openApi/swap/v2/user/balance`
- `positions: /openApi/swap/v2/user/positions`
- `open_orders: /openApi/swap/v2/trade/openOrders`
- `fills: /openApi/swap/v2/trade/allFillOrders`
- `set_margin_type: /openApi/swap/v2/trade/marginType`
- `set_leverage: /openApi/swap/v2/trade/leverage`
- `set_position_mode: /openApi/swap/v1/positionSide/dual`

Источник путей: docs-v3 → USDT-M Perp Futures → Account/Trade Interfaces (см. [[бизнес/инструменты-bingx]] §«Особенности API»).

## 5. Исследование квирков 0.C (что важно проверить на VST)

Это пункты, которые ловим integration-тестами. Расхождения с docs идут в `plans/01` §7 как новые квирки.

1. **`marginType` принимает `ISOLATED`/`CROSSED` (uppercase) — проверить.** В docs-v3 пример пишет именно так. Если BingX молча примет lowercase — фиксируем.
2. **Idempotent `set_margin_type`:** ожидаемое поведение docs — повторный вызов с тем же значением вернёт `code=80012` («No need to switch»). Адаптер должен это ловить и **не** считать ошибкой (метод idempotent).
3. **`set_leverage` per-side:** для one-way режима параметр `side` обязателен (`BOTH`), для dual-side — `LONG`/`SHORT`. В one-way правильное значение — `BOTH`. Проверить на VST.
4. **`set_position_mode` (positionSide/dual)** — параметр `dualSidePosition: true|false` (строкой?). Проверить точный формат на VST.
5. **`/user/balance` формат:** в одних версиях docs `data` — массив, в других — объект `{balance: ...}`. Проверить и зафиксировать модель.
6. **`/user/positions` без аргументов:** возвращает все позиции (включая нулевые) или только открытые? Проверить.
7. **`/trade/openOrders` без `symbol`:** возвращает по всем символам или требует обязательный `symbol`? Проверить.
8. **`/trade/allFillOrders` лимит:** docs указывает максимум `1000`. Проверить, что бьёт по этому лимиту, не больше.
9. **`recvWindow` поведение на больших значениях:** docs говорят максимум `60000` ms. Наш конфиг — 5000 (значение по умолчанию). Проверить что 5000 хватает на нормальный round-trip даже с медленным VPS.
10. **`code=109400` или `code=80014`** — какой именно код BingX отдаёт при timestamp out of recvWindow. По docs — 109400, но реально может оказаться `80014`. Зафиксируем по факту первого падения.

Все эти пункты войдут в integration-тесты как assertion'ы, а расхождения — в §7 plans/01.

## 7. MM-устойчивость

Не применимо как у торговой стратегии. Но **есть один MM-вектор**: если адаптер не делает retry на `109400` (timestamp out of recvWindow), сетевой лаг при volatility-spike приведёт к синхронной серии reject'ов на критичный момент. Митигация: автоматический одноразовый resync + retry в `_send_with_retry` (см. §3).

## 8. 10 причин провала

1. **Ключи утекут в коммит.** Самый страшный сценарий.
   - Митигация: `.env` в `.gitignore` уже стоит. Тесты используют `"test-key"`/`"test-secret"`. `.env.example` — только шаблон. Перед коммитом — `git diff` визуальная проверка. CLAUDE.md запрещает `git add -A` / `git add .` — добавляем файлы поимённо.

2. **Подпись не сойдётся с BingX из-за `params`-порядка / encoding.** Уже было риском в 0.B; smoke на VST это окончательно проверит.
   - Митигация: `sign_query` уже покрыт unit-тестом против docs-вектора. Дополнительный unit с реальной структурой VST-параметров. Smoke `get_balance` — первая точка контроля.

3. **`recvWindow=5000` не хватит на VPS с большим RTT, постоянно `109400`.**
   - Митигация: запустить smoke сначала локально, потом на VPS. Если на VPS падает с 109400 — поднимаем `recv_window_ms` до 10000 в YAML (числа меняются только в YAML, не в коде).

4. **`set_margin_type` повторно бросит ошибку вместо idempotent.**
   - Митигация: ловим `code=80012` («No need to switch») как «успех». Документируем в `private.py` со ссылкой на docs-v3 раздел «Switch Margin Mode» (см. также бизнес/инструменты-bingx.md).

5. **`set_position_mode(one_way)` требует, чтобы не было открытых позиций.** Если на VST случайно есть открытая позиция — вызов отвалится.
   - Митигация: integration-тест предварительно вызывает `get_positions`, и если есть открытые — пропускает (skip) с понятным месседжем. На live этим бы занимался risk-engine pre-flight check (фаза 0.D).

6. **Time-syncer гоняет `/server/time` на каждом вызове** → лишний rate-limit-budget съеден.
   - Митигация: `refresh_if_stale(interval_s=300)` гарантирует не чаще раза в 5 минут. Тест на это.

7. **Decimal-precision: BingX возвращает `"0.00000000"` строкой → pydantic парсит в Decimal с лишним precision, баланс выглядит «0E-8» в логах.**
   - Митигация: Decimal сохраняем как есть (это корректно), но в логи кладём через `format(decimal, "f")` — это уже отдельный задел для 0.E (метрики/логи), здесь только не ломаемся.

8. **VST окружение нестабильное (BingX иногда чинит/ломает demo).**
   - Митигация: integration-тесты помечены `@pytest.mark.integration`, не в CI по умолчанию. Если упали — retry руками, потом repro. Сообщение в issue не нужно — фиксируем как квирк в §7 plans/01.

9. **`set_leverage` максимум зависит от символа** (BTC может иметь до 125x, ETH до 100x и т.д.). Захардкодим 125 — может реджектить.
   - Митигация: локально валидируем 1 ≤ leverage ≤ 125, реальный cap отдаст BingX в response. Метод не пытается «угадать»; правильное плечо для риска приходит из risk-engine (фаза 0.E), не из адаптера.

10. **Утечка ключей через логи.** Если случайно залогируем `request_signed`-params, в логе окажется `signature=...` (это не секрет, но `api_key` через header — тоже не должен светиться).
    - Митигация: при логировании запроса (когда добавим в 0.E) — маскировать `X-BX-APIKEY` и `signature`. Сейчас логирования signed-запросов в коде нет — добавлять только с маскированием.

## 12. Резюме

**Что было:** публичный канал к BingX работает (фаза 0.B), HMAC-подпись реализована как заготовка, но ни одного приватного вызова в коде нет. Нет чтения ключей, нет time-sync, нет моделей `Balance/Position/Order/Fill`, нет setters. Невозможно подготовить аккаунт к ордерам.

**Что стало:** приватный read + idempotent-setters работают на VST. Smoke-сценарий `connect → set_margin → set_leverage → get_balance → disconnect` проходит без падений. Подпись и time-sync проверены на реальном (demo) BingX. Аккаунт VST переключён в `ISOLATED` + `one-way` — пререквизит для ордеров 0.D выполнен.

**Какую проблему решал:** убрал последний слой между «у нас есть код подписи» и «BingX принимает наши signed-запросы». До 0.C любая фаза 0.D начиналась бы с долгой отладки подписи на ордере — а это дороже, чем отладка на `get_balance`, потому что у ордера ещё и побочные эффекты.

**Альтернативы и почему не они:**

- *Сразу делать 0.D (ордера) без 0.C* — отлаживать подпись на ордере означает риск ложного fill'а или зависшего ордера на VST. Дешевле сначала read.
- *Не вынося Settings в отдельный модуль, читать `os.environ` в `BingXClient`* — нарушает разделение слоёв (клиент знает про env), и тестировать сложнее. `pydantic-settings` — стандарт, уже в зависимостях.
- *Делать time-sync фоновым task сразу* — преждевременная сложность. До user-data WS (0.D) хватит ленивого pull при первом signed-запросе + резинк по интервалу.
