# План 33 — Объединённая стратегия: scalp + SMC + DOLF-фильтры

## Решение (однозначное, от владельца 2026-05-16)

Объединяем `scalp_meanrev` (план 31) и `smc_liqsweep` (план 32)
в ОДНУ стратегию `strategies/edge_hybrid/`. Промежуточный гейт по
объёму выборки понижается 100 → 60 OOS-сделок. Стратегию начинаем
использовать (бэктест/demo) и итеративно улучшать. **Цель улучшений
— дорасти до >100 OOS-сделок** и далее проходить полный строгий
гейт. Дополняем элементами DOLF (CVD/OI/ликвидации/funding) как
фильтрами подтверждения.

## Честная рамка (зафиксировано, не для спора — для протокола)

1. Провал 31/32 был не по счётчику, а по статбазе: 4–5 недельных
   корзин (<8) → Sharpe/t не определены. **Легитимная починка —
   именно объединение**: больше сигналов → больше недельных корзин
   → появляется статистика. ≥60 — промежуточная веха, НЕ критерий
   деплоя; всегда печатаем фактическое число недельных корзин.
2. «Использовать» = бэктест + demo. Live скальпинг закрыт
   CLAUDE.md (анти-скальпинг + «без live без demo+явного да»).
   Эти правила безопасности в силе; снятие — только отдельным
   явным подтверждением владельца, не в этом плане.
3. DOLF — по Принципу №1: фильтр/фича после edge-проверки, не
   триггер.
4. Анти-оверфит (AGENTS.md): параметры из логики; понижение гейта
   компенсируется честным walk-forward и cost-sweep, которые НЕ
   ослабляются.

## Архитектура edge_hybrid

Одна стратегия, два независимых генератора сигналов + общий
DOLF-фильтр и общий риск/выход.

**A. Mean-reversion ветка (из scalp_meanrev):**
- |close − EMA(anchor)| ≥ entry_k·ATR → ставка на возврат.
- Фильтр сильного дневного тренда |EMAf−EMAs|/EMAs > trend_block.

**B. Liquidity-sweep ветка (из smc_liqsweep):**
- Прокол swing-экстремума `L` баров хвостом ≥ sweep_k·ATR + reclaim
  телом обратно за уровень → reversion против свипа.
- Тот же фильтр тренда.

Сигнал = A ∨ B (любая ветка). Это удваивает частоту входов →
цель статбазы. Конфликт A и B в одном баре: приоритет B (свип —
более структурное событие), лог обоих.

**C. DOLF-фильтр подтверждения (Принцип №1, фильтр не триггер):**
- Доступные элементы из `core/signals`: detect_order_flow (CVD),
  detect_oi_trend (OI), detect_liquidation_sweep, detect_funding_extreme.
- Применение: вход A/B пропускается ТОЛЬКО если DOLF-контекст
  НЕ противоречит развороту (напр. для BUY-reversion: нет
  свежего бычьего OI-разгона против нас; либо ликвидационный
  свип вниз подтверждает вымывание). По умолчанию фильтр
  «мягкий» (блокирует явное противоречие), не обязательное И.
- **Ограничение данных (честно):** Coinglass HOBBYIST даёт 15m
  DOLF лишь ~180д; полная история — daily/4h. v1: DOLF как
  старший-ТФ (4h/1d) режимный фильтр, состыкованный по времени;
  где интрадей-DOLF нет — фильтр выключен (логируется), вход по
  чистой цене. Рост покрытия DOLF — в roadmap улучшений.

**Выход/риск:** общий. SL = entry ± max(sl_k·ATR, stop_min%).
TP: ветка A → якорь EMA; ветка B → tp_r·R. RiskEngine как в 31/32.

## Гейт (промежуточный → целевой)

Промежуточный (этот план): OOS PF>1.3 ∧ Sharpe>0.8 ∧ t>1.5 ∧
**≥60 сделок** ∧ ≥8 недельных корзин; IS≈OOS; cost-sweep
0.10/0.15/0.20% — НЕ ослабляется; бьёт B&H.

Целевой (куда растём, не ослабляется): OOS PF>1.5 ∧ Sharpe>1.0
∧ t>2 ∧ **≥100 сделок**; всё остальное как было (план 31).

Любой провал cost-sweep → отклонить независимо от счётчика.

## 10 причин провала (априорно, честно)

1. Объединение коррелированных reversion-веток не добавляет
   независимых корзин (A и B ловят то же движение).
2. ≥60 наберём, но недельных корзин всё равно <8 → стат-нуль.
3. DOLF старшего ТФ слишком грубый для 15m-входа (бесполезен).
4. cost-sweep убивает (главный killer скальпа, не изменилось).
5. Понижение PF/Sharpe-порога маскирует отсутствие edge.
6. Селекция: 2 сетапа из многих → multiple testing, нужен WF.
7. IS≠OOS: режимная зависимость reversion.
8. Расширение юниверса = скрытая подгонка, если post-hoc подбор.
9. Интрадей-DOLF нет → фильтр почти всегда выключен → толку нет.
10. Даже пройдя промежуточный гейт — это НЕ edge, а лишь право
    продолжать исследование (явно коммуницировать владельцу).

## 33.8 РЕДИЗАЙН по требованию владельца — настоящая частота

Владелец: «стратегия нужна, доработай чтобы работала, других
данных нет». Делаю инженерный редизайн ПРИЧИНЫ (не подгонку).

**Диагноз причины (точный, по коду движка):** движок игнорирует
сигнал при открытой позиции (engine.py:162); выходы — только
attached SL/TP покадрово или конец теста. Средняя сделка ~8
баров — позиции НЕ висят неделями. Редкость и кучкование — от
ВХОДА: фильтр тренда требует ШТИЛЬ (EMA-spread ≤1.5%), а ветка A
требует отклонение 2·ATR (~2σ). Эти условия почти не
сочетаются; когда сочетаются — это одно и то же рыночное
событие → 3–4 коррелированных окна.

**Редизайн (логика, НЕ оверфит, cost-sweep НЕ ослабляю):**
1. Ветка A → настоящий частый интрадей-mean-reversion:
   отклонение от короткого якоря на ~1·ATR (учебный скальп-порог
   ~1σ, не подбор под бэктест), тугой SL, TP=якорь (быстрый
   возврат). 1σ-снапбэки происходят регулярно во всех режимах →
   сделки размазываются по многим неделям (прямая атака на
   корень 3–4 корзин).
2. Снять запретительный «только штиль»: вместо жёсткого блока —
   мягкий фильтр лишь сильнейшего тренда (высокий порог), т.к.
   мелкая реверсия бывает в любом режиме. Контр-тренд только при
   умеренном рынке; в сильном тренде A выключается, остаются B/C.
3. B (свип) и C (пробой) сохраняются как есть — для трендовых
   окон, дают входы там, где A молчит.
4. Параметры — из торговой логики (1σ вход, ATR-стоп, R-цель),
   НЕ скан «лучшего» по Sharpe. Я НЕ обещаю профит — обещаю
   статистически проверяемую частоту и честный отчёт.

**Гейт без изменений:** промежуточный (PF>1.3 Sh>0.8 t>1.5 ≥60
∧ ≥8 нед.корзин) и целевой (≥100); cost-sweep 0.10/0.15/0.20%
и WF НЕ ослабляются. Прогон по 5m/15m/30m.

**Жёсткие стопы те же:** анти-скальпинг CLAUDE.md в силе —
бэктест/demo, НЕ live без отдельного явного «да». «Прошёл» =
право продолжать, не edge и не деньги.

## 33.12 ПЕРВЫЙ ПОЛОЖИТЕЛЬНЫЙ СИГНАЛ — funding-extreme + liq

### 33.12 walk-forward — лид НЕ подтвердился (артефакт сплита)
