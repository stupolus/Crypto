# Деплой 24/7: VPS + systemd + Telegram + healthcheck

**Цель:** бот крутится без участия Claude и пользователя. Падает — поднимается сам. Сбой — приходит Telegram-алерт.

**Стоимость:** ~$5-10/мес VPS + 0 на остальное.

**Время на установку с нуля:** 30-40 минут когда у тебя есть SSH-доступ.

---

## 1. Что нужно от тебя

### 1.1 VPS

**Где взять:**
- **Hetzner** (`hetzner.com/cloud`) — Сингапур/Гонконг $5/мес, 2 GB RAM (`CPX11`). Близко к BingX серверам.
- **DigitalOcean** — Singapore datacenter, $6/мес basic droplet.
- **Vultr** — Tokyo/Singapore, $6/мес.

**Что выбрать:** Hetzner CPX11 в Сингапуре. Соотношение цена/latency лучшее для BingX.

**ОС:** Ubuntu 24.04 LTS.

### 1.2 Telegram бот

1. В Telegram открыть [@BotFather](https://t.me/BotFather).
2. `/newbot` → дать имя → получить **`TELEGRAM_BOT_TOKEN`** (формат `123456789:ABCdef...`).
3. Написать своему боту любое сообщение в Telegram (открыть чат с ботом).
4. В браузере: `https://api.telegram.org/bot<TOKEN>/getUpdates`.
5. Найти в ответе `"chat":{"id":12345...}` — это **`TELEGRAM_CHAT_ID`**.

### 1.3 BingX API ключи (если ещё нет)

Уже есть VST-ключи в `.env`. Для live — отдельные ключи с **только торговыми правами** (без вывода) и IP whitelist на IP VPS.

### 1.4 SSH-доступ к VPS

Закинуть SSH-ключ (`ssh-copy-id` или через панель провайдера).

## 2. Что делает Claude (один раз)

### 2.1 `scripts/deploy/install.sh`

Идемпотентный скрипт установки на свежей Ubuntu:
- `apt-get install python3.12 python3.12-venv git`
- Создаёт пользователя `crypto`
- Клонирует репо в `/opt/crypto`
- `python3.12 -m venv /opt/crypto/.venv`
- `pip install -e ".[dev]"`
- Создаёт `/etc/crypto/.env` (chmod 600) с шаблоном

### 2.2 `scripts/deploy/crypto-runner@.service`

systemd unit с шаблоном на символ:

```ini
[Unit]
Description=Crypto bot — %i runner
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=crypto
WorkingDirectory=/opt/crypto
EnvironmentFile=/etc/crypto/.env
ExecStart=/opt/crypto/.venv/bin/python -m runners.live_runner \
    --strategy btc_breakout \
    --symbol %i \
    --interval 15m \
    --warmup-candles 100 \
    --metrics-file /var/lib/crypto/%i-metrics.jsonl \
    --journal-db /var/lib/crypto/%i.sqlite \
    --log-level INFO

# Автоматический рестарт при крэше.
Restart=on-failure
RestartSec=10s
StartLimitInterval=300
StartLimitBurst=5

# Logs: journald (просмотр: journalctl -u crypto-runner@BTC-USDT -f)
StandardOutput=journal
StandardError=journal
SyslogIdentifier=crypto-%i

# Безопасность: read-only / минимум прав.
ProtectSystem=strict
ReadWritePaths=/var/lib/crypto
NoNewPrivileges=true

[Install]
WantedBy=multi-user.target
```

**Запуск раннеров:**
```bash
sudo systemctl enable --now crypto-runner@BTC-USDT
sudo systemctl enable --now crypto-runner@ETH-USDT
sudo systemctl enable --now crypto-runner@XRP-USDT
```

**Логи:**
```bash
journalctl -u crypto-runner@BTC-USDT -f
journalctl -u 'crypto-runner@*' --since "1 hour ago"
```

**Auto-restart при крэше:** systemd рестартит до 5 раз за 5 минут. Если падает чаще — escalation (см. ниже).

### 2.3 `scripts/deploy/log-rotation.conf` (`/etc/logrotate.d/crypto`)

```
/var/lib/crypto/*.jsonl /var/lib/crypto/*.log {
    daily
    rotate 14
    compress
    delaycompress
    missingok
    notifempty
    copytruncate
}
```

Хранит 14 дней метрик, остальное архивирует с gzip.

### 2.4 `scripts/deploy/healthz.py`

Маленький HTTP endpoint (FastAPI или встроенный):
- Слушает `:8080/healthz`
- Возвращает 200 если все systemd-units `crypto-runner@*` в статусе `active (running)`
- Возвращает 503 если хоть один не запущен

UptimeRobot пингует каждые 5 мин → если 2 раза подряд 503 → email/Telegram алерт.

## 3. Эскалация при проблемах

### 3.1 Раннер падает повторно (>5 раз за 5 мин)

systemd прекращает рестарт → unit в статусе `failed` → healthz возвращает 503 → UptimeRobot шлёт алерт.

**Что делать:** SSH на VPS, `journalctl -u <unit> --since "10 min ago"`, читать.

### 3.2 BingX API дал auth error 5+ раз подряд

Runner вызывает `alerter.send_critical("AuthError")` через Telegram → приходит на телефон.

**Что делать:** проверить .env (ключи не протухли), IP whitelist на BingX.

### 3.3 Daily stop сработал (-3% эквити)

Runner вызывает `alerter.send_critical("daily stop reached")` → приходит в Telegram → бот выключается до следующего дня (через kill switch адаптера).

### 3.4 Сеть на VPS лежит

UptimeRobot фиксирует через 5 мин → шлёт алерт на email + Telegram.

## 4. Что ВСЁ ЕЩЁ нужно от пользователя

После 24/7 деплоя:
- **Раз в неделю**: смотреть отчёт по D3 (статус-скрипт + ручной анализ trades).
- **Раз в месяц**: ретроспектива в `retro/`.
- **При сигналах что-то новое**: новая фича = новый план в `plans/` = Claude делает.

**НЕ надо**: каждые 15 минут заходить и проверять. **Бот сам.**

## 5. План реализации (когда у тебя будет VPS)

| Шаг | Делает | Время |
|-----|--------|-------|
| 1. Купить VPS, получить SSH | Пользователь | 10 мин |
| 2. Создать Telegram бота | Пользователь | 5 мин |
| 3. Запустить `install.sh` на VPS | Claude (через ssh) | 10 мин |
| 4. Положить `.env` (BingX + Telegram ключи) | Пользователь | 2 мин |
| 5. `systemctl enable crypto-runner@<SYM>` × 3 | Claude | 1 мин |
| 6. Поднять healthz + UptimeRobot мониторинг | Claude + пользователь (UR signup) | 10 мин |
| 7. Проверить что Telegram-алерт приходит | Claude (форс fake alert) | 2 мин |

**Итого ~40 минут от пустого VPS до 24/7.**

## 6. Что НЕ делается в этом плане (отложено)

- **Multi-VPS redundancy** — два VPS с failover. Фаза 3+, когда депозит > $10k.
- **Prometheus + Grafana дашборд** — позже. Сейчас достаточно `journalctl` + `d3_status.py` + Telegram.
- **Docker контейнеры** — overkill. systemd проще, надёжнее, быстрее на VPS этого размера.
- **CI/CD auto-deploy** — позже. Сейчас деплой = `git pull && systemctl restart`.

---

## 7. Безопасность

- `.env` файл с ключами — `chmod 600`, владелец `crypto`. Никто кроме него не читает.
- `crypto` пользователь без sudo. Не может править systemd / читать другие файлы.
- VPS firewall (`ufw`): только SSH (22) + healthz (8080 ограничен по IP UptimeRobot).
- BingX API ключи — **только торговые** (без вывода), IP whitelist на IP VPS.
- SSH — только по ключу, password auth выключен.
- `fail2ban` для brute-force защиты SSH.

---

## 8. Резервная копия

Каждые 6 часов crontab делает дамп `/var/lib/crypto/` в S3/Backblaze (опционально). Это journal + metrics — критично для восстановления состояния после крэша.

**Команда (cron):**
```bash
0 */6 * * * tar -czf /tmp/crypto-$(date +\%Y\%m\%d\%H).tar.gz /var/lib/crypto/ && rclone copy /tmp/crypto-*.tar.gz backup:crypto/
```
