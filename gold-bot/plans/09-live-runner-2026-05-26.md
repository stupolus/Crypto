# 09 — Live-runner (реальные ордера) — DESIGN ONLY, код после явного «да»

Дата: 2026-05-26
Статус: design / черновик. **Кода в этом коммите нет.** Реализация
начнётся только после: (а) 4+ недель paper-наблюдения с зелёными метриками
по плану 06, (б) PASS champion-challenger по плану 08, (в) явного письменного
«да» пользователя на конкретную стратегию.

## Цель

Поднять реальную интрадей-торговлю на BingX (USDT-perp) на стратегии,
прошедшей все предыдущие гейты. Активный банк — **строго $1000–$2000**
(фаза 2 из risk-profile.md). Любая попытка стартануть на большем
капитале без отдельного решения — провал инварианта.

## Что обязано работать ДО первого реального ордера

- 4+ недели paper-журнала в SQLite на VPS с PF ≥ 1.2 и max DD ≤ 6%.
- API-ключи с правами **только trade + read, без права вывода средств**.
- IP whitelist на стороне биржи — VPS-IP единственный.
- Bybit testnet integration-тесты зелёные (плана 01) — sanity check адаптера.
- Sync-positions-on-start проверен на bybit testnet — никаких дублей.
- Watchdog (план 10) — если runner молчит >5 минут, alert + market-close.

## Архитектура

```
live/
├── runner.py        # LiveRunner: тот же контракт, что paper, но real adapter
├── safeguards.py    # все гейты старта: env, paper-history, allowlist, kill-switch
├── reconciler.py    # sync_positions_on_start: читает биржу, согласует journal
└── tests/           # integration на bybit testnet, обязательно
```

`LiveRunner = PaperRunner` с заменой:
- adapter: `BingXAdapter(api_key, api_secret, vst=False)` или Bybit prod.
- engine: вместо «виртуального fill» — реальный `place_order` с attached SL.
- exit: вместо «check high/low бара» — наблюдение реального исполнения
  через `fetch_open_orders` + `watch_my_trades` (если адаптер поддерживает).

## Гейты старта (запекаются в `safeguards.py`, не опциональны)

Порядок проверки на `LiveRunner.start()`:

1. **`GOLDBOT_LIVE_CONFIRM` env**: точная строка
   `I-UNDERSTAND-CAPITAL-AT-RISK-<DATE>` где DATE = сегодня UTC. Иначе
   `RuntimeError`. Защита от случайного запуска вчерашним env.
2. **Стратегия в allowlist**: `config/live.yaml::active_strategy` ∈
   `{mean_reversion_vwap.v1, ...явный список из коммита}`. Любая стратегия
   не из списка — `RuntimeError`.
3. **Paper-история**: открываем `/var/lib/gold-bot/paper.sqlite`, читаем
   `daily_summary` за последние 28 дней:
   - ≥ 28 закрытых дней с торговлей
   - агрегатный PF ≥ 1.2
   - агрегатный max DD ≤ 6%
   - сумма trades ≥ 100
   Иначе `RuntimeError`.
4. **Active capital ≤ phase limit**: читает Balance с биржи, проверяет
   что `total_usdt ≤ phase_max` из `risk-profile.md` (для фазы 2 = $2000).
   Иначе `RuntimeError`.
5. **No live withdrawal rights**: вызывает `fetch_balance` и проверяет
   что в ответе нет прав на withdraw (если API даёт такую информацию;
   если нет — pre-flight требует ручного аттеста через env).
6. **Single instance lock**: PID-файл в `/var/lib/gold-bot/live.pid`,
   exclusive flock. Два LiveRunner'а одновременно — RuntimeError.

Если любая проверка падает — runner не стартует. Никаких «warn and continue».

## Reconciliation на старте

`reconciler.sync_positions_on_start()`:
1. `fetch_positions()` с биржи.
2. Сравнить с `runner_state.open_position:*` в SQLite.
3. Случаи:
   - **Биржа: позиция есть, journal: нет.** Журналируем как «recovered»,
     ставим stop по ближайшей разумной цене (best ask/bid ± k_stop·ATR),
     включаем мониторинг. Alert в Telegram.
   - **Биржа: нет, journal: есть.** Удаляем из journal как «closed externally».
     Alert в Telegram.
   - **Обе есть, но qty/price расходятся.** Биржа = истина. Перезаписываем
     journal. Alert.
4. После reconciliation runner стартует с чистой согласованной картой мира.

## Circuit breakers — реальные действия

В paper-runner'е circuit breakers — это halted_days в RiskState. В live:
- daily_stop: cancel-all + market-close всех позиций + halt до 00:00 UTC.
- kill-switch (−15% от пика): то же + Telegram-CRITICAL + остановка процесса.
- consecutive losses: market-close + halt до 00:00 UTC.
- WS-disconnect > 30 сек: market-close (по risk-profile.md).

## Telegram-alerting

- На вход в позицию (с qty, entry, stop, tp, expected risk_amount).
- На выход (с net_pnl, equity_after).
- На любой `RuntimeError` safeguard.
- На любой `RuntimeError` reconciler.
- Daily summary (то же что paper).
- При срабатывании любого circuit breaker — CRITICAL.

## Что НЕ делаем в этом плане

- Bybit live. Только BingX. Bybit testnet — для integration-тестов адаптера.
- Multi-symbol arbitrage между биржами.
- Champion-challenger в live. Live = только champion из конфига.
- Hedge-mode / multi-position по одному символу. CLAUDE.md §6: одна позиция.

## 10 причин почему может не получиться

1. **BingX VST не отражает live-поведение** (slippage, latency). Решение:
   фаза 2 малая сумма ($1k) на live — это и есть первая реальная проверка.
2. **Attached stop-loss через ccxt не работает на BingX как ожидаемо.**
   Решение: integration-тесты на bybit testnet до live-старта; fallback —
   отдельный stop-ордер сразу после fill.
3. **Реконсилер криво интерпретирует hedged-position.** Решение: на старте
   проверяем что position-mode = one-way, иначе RuntimeError.
4. **Сеть VPS — провал во время открытой позиции.** Решение: heartbeat
   watchdog на отдельном процессе (план 10), market-close при таймауте.
5. **Time-skew VPS vs биржа > 1 сек.** Решение: NTP-проверка на старте,
   `fetch_time()` сверить с `time.time()`, отказ при > 1 сек.
6. **Биржа меняет precision символа (PAXG/USDT:USDT)** и наши Decimal-расчёты
   возвращают ордер с лишним знаком. Решение: на старте `fetch_markets` и
   нормализация qty/price по precision символа.
7. **Дублирующий запуск runner'а после ручного рестарта.** Решение: flock
   PID-файла.
8. **Telegram упал → kill-switch не алертит.** Решение: алерт пытается
   написать также в `runner_state.last_alert_failed` и журналит в stdout;
   docker logs / journalctl всё равно содержат событие.
9. **API-ключ с правом withdraw.** Решение: safeguards.requires_no_withdraw()
   проверяет env-аттест от оператора (`GOLDBOT_LIVE_NO_WITHDRAW=true`) и
   предупреждает что биржевые API могут не сообщать это honestly.
10. **Champion стратегия перестала иметь edge между paper и live.** Это
    основная причина фазы 2 с малым капиталом. Если paper PF ≥ 1.2, а live
    PF < 0.8 за неделю — runner сам себя останавливает (новый circuit
    breaker `live_vs_paper_divergence`).
