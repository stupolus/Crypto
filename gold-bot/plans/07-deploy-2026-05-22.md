# 07 — Деплой paper-runner'а на VPS (Docker + systemd + ежедневный отчёт)

Дата: 2026-05-22
Статус: первичный, готов к реализации

## Цель

Дать paper-runner'у способ стабильно жить на VPS: автоперезапуск при падении,
переподключение к API при обрыве, ежедневный отчёт в Telegram. Один способ
запуска (Docker) и один резервный (systemd напрямую), оба одинаково
безопасны и не открывают пути к live-торговле.

## Не цели

- Live-торговля. План 07 не позволяет её запустить (только paper).
- Дашборд / веб-UI. Логи + Telegram достаточно на фазе 1.
- Multi-host / HA. Один VPS — один runner.

## Архитектура

```
gold-bot/deploy/
├── Dockerfile             # python:3.12-slim + deps + код, ENTRYPOINT paper
├── docker-compose.yaml    # один сервис gold-bot-paper, bind /var/lib/gold-bot
├── systemd/
│   └── gold-bot-paper.service  # резервный systemd-юнит (без Docker)
└── README.md              # как развернуть, как остановить, как смотреть логи

gold-bot/scripts/
└── daily_report.py        # печатает daily summary + шлёт в TG; запуск из cron @ 00:05 UTC
```

## Инварианты деплоя

1. **Никаких live-флагов.** Образ собирается без `BINGX_LIVE_API_KEY` /
   `BYBIT_LIVE_API_KEY` в окружении. Даже если кто-то добавит — `run_paper.py`
   не вызывает приватные методы биржи.
2. **Журнал SQLite — на хост-volume.** `/var/lib/gold-bot/` (на VPS) →
   `/var/lib/gold-bot/` (в контейнере). Обновление кода не теряет историю.
3. **Логи — stdout JSON.** Docker и systemd журналируют → `journalctl -u`
   или `docker logs`. Никаких ротируемых файлов внутри контейнера.
4. **Автоперезапуск.** Docker `restart: unless-stopped`; systemd
   `Restart=on-failure`. Краш = перезапуск через 10 сек.
5. **Стартовая синхронизация.** PaperEngine при создании читает journal:
   equity, open_position, last_candle_ts. Перезапуск = бесшовное продолжение
   без дублей сделок (контракт уже реализован в плане 06).
6. **Time-drift.** VPS обязан иметь NTP (системный, вне gold-bot). Docker
   наследует часы хоста.
7. **Секреты только через env.** `.env` на хосте, права 600,
   `env_file: .env` в compose. В образ не попадают.

## Поток деплоя (для оператора)

1. На VPS клонировать репозиторий и переключиться на ветку `gold`.
2. Создать `/var/lib/gold-bot/` (mkdir, owner = root, mode 750).
3. Создать `.env` рядом с docker-compose.yaml с `GOLDBOT_TG_TOKEN` и
   `GOLDBOT_TG_CHAT_ID` (опционально). Без них — runner всё равно работает.
4. `docker compose -f gold-bot/deploy/docker-compose.yaml up -d`.
5. `docker compose logs -f` для наблюдения.
6. Cron на хосте: `5 0 * * * docker exec gold-bot-paper python -m scripts.daily_report`
   или эквивалент в systemd timer (см. README).

## Ежедневный отчёт

`scripts/daily_report.py`:
- Считает суммарные метрики за прошедший UTC-день из SQLite.
- Формирует текст: trades, winrate, PF, day_pnl, equity, статус
  circuit breakers, heartbeat.
- Печатает в stdout + шлёт через Telegram-нотификатор (если env есть).
- Записывает summary в таблицу `daily_summary` SQLite.

Триггер: cron @ 00:05 UTC на хосте. Можно перевести в внутренний asyncio-таск
позже — на v1 cron проще и надёжнее.

## Что подразумевается «зелёным» этим планом

- `docker compose up -d` поднимает runner, тот ставит heartbeat, не падает 24 часа.
- `daily_report` присылает осмысленный текст (даже если 0 сделок —
  `0 trades, equity=10000`).
- После перезапуска контейнера: equity и open_position те же, журнал
  не дублируется.

## Что НЕ покрывает план

- Live-runner (план 09, после ≥4 недель paper).
- Champion-challenger (план 08, после первого месяца paper).
- Алертинг на инциденты типа «runner молчит >N минут» — это отдельная
  мониторинг-фаза; в текущем плане heartbeat пишется в SQLite, watchdog
  обещаем добавить позже.

## 10 причин почему может не получиться

1. **Docker нет на VPS.** Решение: systemd-юнит как резерв (`deploy/systemd/`).
2. **VPS не имеет NTP.** Свеча кажется закрытой раньше времени → ложные
   сигналы. README требует `timedatectl set-ntp true`.
3. **SQLite-файл создаётся root'ом в контейнере** → нельзя писать с хоста.
   Решение: контейнер запускается под uid 1000, /var/lib/gold-bot
   chown 1000:1000.
4. **Утечка env в `docker inspect`.** Решение: env_file, не `environment:`
   с открытыми значениями.
5. **Образ собирается с кешем pip → новые зависимости не подтягиваются.**
   Решение: `--no-cache-dir` в Dockerfile.
6. **Конфликт `pyproject.toml` package discovery** (уже видели локально).
   Решение: Dockerfile не делает `pip install -e .`, ставит зависимости
   списком, код копируется как есть.
7. **Cron `daily_report` стартует когда runner ещё не дописал сделку
   в SQLite.** Решение: транзакции (есть), запуск daily_report в 00:05
   (после ротации UTC-дня).
8. **Telegram падает / rate-limit.** Reporter возвращает False, runner
   не падает. Daily-report повторит на следующий день.
9. **Контейнер перезапускается слишком часто → spam Telegram'а.**
   Решение: в `run_forever` уведомление о старте/остановке debounce
   по rate-limit reporter'а (уже есть, max 30/час).
10. **Хост уходит на reboot, контейнер не поднялся.** Решение:
    `restart: unless-stopped`; альтернатива systemd `Restart=always` + `WantedBy=multi-user.target`.

## Критерии приёмки

- `docker build -f gold-bot/deploy/Dockerfile gold-bot/` собирается локально (на VPS).
- `docker compose up -d` поднимает контейнер, `docker compose logs` показывает
  `paper.warmup` + heartbeat'ы.
- `python -m scripts.daily_report` (внутри контейнера или хоста) печатает summary.
- systemd-юнит загружается (`systemctl daemon-reload && systemctl enable gold-bot-paper`).
