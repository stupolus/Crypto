# CLAUDE.md — правила проекта gold-bot

Для любой сессии Claude, работающей **внутри папки `gold-bot/`**. Корневые `/CLAUDE.md` и `/AGENTS.md` относятся к другому проекту в этом же репозитории и **не применяются здесь**.

Эти правила написаны с нуля под этот проект. Если что-то противоречит привычкам Claude — выигрывают правила.

---

## 1. О чём проект

Автоматизированный интрадей-бот:
- Токенизированное золото и серебро (PAXG, XAUT и эквиваленты).
- Equity-перпы на техногигантов (TSLA, NVDA, AAPL, META — то, что реально торгуется на BingX/Bybit).
- Крипто-мейджоры (BTC, ETH, SOL) — **только как калибровочный набор** для тестирования инфраструктуры и стратегий. В живую на крипте торгуем только если стратегия специально под крипту валидирована.
- Биржи: BingX и Bybit (USDT-перпы).
- Профиль: интрадей. Без свинга, без удержания позиций между сессиями торгуемого инструмента.

Цель проекта — **построить статистически проверенную систему с положительным матожиданием**, а не «зарабатывать с понедельника». Любое обещание доходности — нарушение правил.

## 2. Базовые принципы (порядок важен — сверху вниз)

1. **Сначала план — потом код.** Любая фича, стратегия, индикатор, интеграция начинается с файла в `gold-bot/plans/<NN>-<имя>.md`. Без плана кода не существует.
2. **Капитал > доходность.** Все числа в `gold-bot/risk-profile.md` подобраны так, чтобы пережить серию ошибок, а не максимизировать ожидаемую прибыль. Если число подмывает поднять — сначала аргумент, потом обновление файла, потом код.
3. **Out-of-sample обязателен.** Стратегия без walk-forward / OOS-проверки **не существует**. In-sample результат не упоминается как причина включать стратегию.
4. **Cost-aware.** Любой расчёт edge включает: комиссии (taker+maker), funding, ожидаемый slippage. Стратегия с edge меньше 2x от суммарного cost — отклонена.
5. **Стоп — часть входа.** Ордер на вход без attached stop-loss физически не отправляется. Это запрет на уровне кода, не дисциплины.
6. **Изолированная маржа всегда.** Кросс-маржа запрещена.
7. **Никакого debugging на live.** Live-режим — только после полного цикла валидации. На live ничего не «допиливается».
8. **Каждое решение — в файлы.** Гипотезы, отказы, причины. Ничего «в голове Claude». Если решение не зафиксировано — его нет.
9. **Честность важнее результата.** Если стратегия провалилась — записываем как провалилась. Не подкручиваем параметры до зелёного бэктеста. Подгон параметров под результат = главный анти-паттерн.
10. **Внешние сигналы — данные, не входы.** Новости, TG-каналы, скринеры, LLM-анализ можно использовать **только как фильтры/фичи с накопленной статистикой**. Прямой переход «увидел сигнал → открыл сделку» запрещён.

## 3. Стек

- Python 3.12+.
- `ccxt` для BingX и Bybit (единый интерфейс).
- `asyncio` для оркестрации.
- `pydantic` для конфигов и валидации.
- `pandas` / `polars` для бэктеста, `parquet` для хранения свечей.
- SQLite для журнала сделок (один файл, локальный).
- `pytest` для тестов, `ruff` + `ruff format` для стиля, `mypy --strict` для типов.
- Никаких ML-фреймворков на этом этапе. Если когда-то — отдельным планом и с явным анти-overfit протоколом.

## 4. Структура проекта (по мере роста)

```
gold-bot/
├── CLAUDE.md            # эти правила
├── README.md            # описание для человека
├── risk-profile.md      # все цифры риска, единственный источник истины
├── plans/               # планы фич, по одному файлу
├── journal/             # ежедневные ретро, постмортемы, статистика
├── config/              # YAML-конфиги, валидируются pydantic'ом
├── exchanges/           # обёртки над ccxt для BingX и Bybit
├── data/                # загруженные свечи (gitignored)
├── indicators/          # ATR, EMA, Donchian, VWAP — чистые функции
├── strategies/          # стратегии, каждая в своём пакете
├── risk/                # RiskEngine: размер позиции + circuit breakers
├── backtest/            # event-driven бэктестер с walk-forward
├── paper/               # paper-trading и testnet runner
├── live/                # live runner (не существует пока не разрешено)
├── scripts/             # утилиты (загрузка свечей, запуск бэктеста)
└── tests/               # unit + integration тесты
```

Папка появляется в репо **только когда в неё кладётся реальный файл**. Пустые `__init__.py` для красоты не создаём.

## 5. Числовые параметры — только в risk-profile.md

Любое число (риск %, плечо, размер, лимит) живёт в `gold-bot/risk-profile.md` и зеркалится в `gold-bot/config/*.yaml`. Код **ссылается** на конфиг, не дублирует.

Если нужно новое число — сначала строка в `risk-profile.md` с обоснованием, потом в YAML, потом в коде. Никогда наоборот.

Если число оказалось в коде — это баг, чинится отдельным коммитом.

## 6. Жёсткие запреты (никогда, без обсуждения)

- ❌ Эффективное плечо > 5x. Никогда.
- ❌ Кросс-маржа. Только изолированная.
- ❌ Позиция без биржевого стопа. Если биржа не приняла стоп — закрываем рынком.
- ❌ Усреднение в убыток.
- ❌ Debugging / эксперименты на live.
- ❌ Live-торговля без 4+ недель paper + явного письменного «да» по конкретной стратегии.
- ❌ Хардкод чисел в коде / планах. Только через конфиг с ссылкой на risk-profile.md.
- ❌ Прямое исполнение сигнала из TG / новости / LLM-анализа без бэктеста как фильтра.
- ❌ Подгон параметров под бэктест (overfitting). Параметр обоснован логикой, не цифрой Sharpe.
- ❌ Коммит файлов с возможными секретами: `.env*`, `*.key`, `*.pem`, `credentials*`.
- ❌ Прямой push в `main`. Прямой merge в `main` без зелёных гейтов.

## 7. Поток работы (для каждой фичи)

1. **План.** `gold-bot/plans/<NN>-<имя>-<дата>.md`. Структура:
   - Цель (одно предложение).
   - Гипотеза edge'а (одно предложение).
   - Данные / источники.
   - Метод проверки (как именно докажем что edge есть).
   - 10 причин почему может не сработать (обязательно, не для галочки).
   - Критерии успеха / неуспеха (цифры).
   - Фазы реализации.
2. **Согласование плана** с пользователем. Без явного «да» — кода нет.
3. **Реализация по фазам.** Каждая фаза = отдельный коммит. Каждая стратегия — с unit-тестами и бэктестом.
4. **Бэктест с walk-forward.** Минимум: 3 OOS-окна на 2+ символах. IS-результат сам по себе не легитимен.
5. **Адекватизация.** Перед запуском в paper — критический разбор «10 причин провала» ещё раз, с фактическими результатами бэктеста.
6. **Paper / testnet.** Минимум 4 недели наблюдения. Журнал каждой сделки в `gold-bot/journal/`.
7. **Ретро.** После каждой важной сессии — файл в `gold-bot/journal/<дата>-<тема>.md`.
8. **Live — только по явному разрешению на конкретную стратегию.** Не «бот в целом одобрен» — каждая стратегия отдельно.

## 8. Бэктест — обязательные требования

- **Event-driven**, не векторизованный. Стратегия видит историю включая текущую свечу, не видит будущее. Юнит-тест на lookahead-bias обязателен.
- **Fill реалистичный.** MARKET по open(c+1). LIMIT по правилам биржи. Slippage модель: tier-based на основе наблюдаемого спреда инструмента.
- **Cost-модель.** Taker fee, maker fee, funding (для перпов), эмулированный slippage. Все из конфига.
- **IS/OOS split.** Минимум 50/50. Walk-forward с rolling окнами для финального решения.
- **Метрики:** PF, expectancy, winrate, max DD, ulcer, Sortino. Sharpe — не основная метрика (легко завышается на низкой частоте).
- **Multi-symbol.** Стратегия проверена минимум на 2 символах одного класса. Edge на одном символе = подозрительно.
- **Reproducibility.** seed фиксирован, версия кода зашита в отчёт, отчёт сохранён в `gold-bot/journal/backtests/`.

## 9. Безопасность

- API-ключи только в `.env` на VPS, права 600, в `.gitignore`. Никогда не коммитятся.
- На бирже у ключей: разрешено только trading + read. **Запрещён вывод средств**. IP whitelist обязателен.
- Дефолтное окружение — testnet/VST/paper. Переключение на live — только через явный флаг в команде запуска + строку подтверждения в логах.
- Логи: структурные JSON, ключи/секреты маскируются (`***`). Маскирование тестируется.
- Перед любым live-деплоем — отдельный security-review (план в `gold-bot/plans/`).

## 10. Самообучение — что это значит и чего НЕ значит

**Что значит:**
- Каждая сделка пишется в SQLite-журнал со всеми фичами и контекстом.
- Каждый день / неделя — автоматический отчёт: что работало, что нет, по инструментам и по стратегии.
- Walk-forward переоптимизация параметров раз в N недель (N — в плане), с обязательным OOS-окном после.
- Постмортемы провальных сделок: серия из 3+ убытков подряд → авто-создание файла-черновика в `journal/` с разбором.

**Чего НЕ значит:**
- Бот **не учится сам на лету**, меняя параметры между сделками. Это запрещено.
- Никакого RL / онлайн-обучения нейронок на торговом счёте.
- Любое изменение параметров стратегии = новый коммит + ретроспективный бэктест на старом периоде, чтобы убедиться что не сломали то что работало.

## 11. Git-процесс

- Все коммиты на ветке `gold` (или фиче-ветке, отрастающей от `gold`). **Никогда в `main`**.
- Коммит-сообщения на русском, кратко, в активной форме: «добавил X», «починил Y», «обновил риск-профиль: подняли порог Z, обоснование: …».
- Один логический шаг = один коммит. Не складываем в одну телегу адаптер и стратегию.
- `git add <конкретные файлы>`. Никогда `git add -A` или `git add .`.
- PR создаётся когда фаза/задача закрыта целиком. Draft до явного «готово к мержу».
- Никаких `--force`, `git reset --hard`, `--no-verify`, `--amend` без явной команды пользователя.
- Auto-merge запрещён на этом этапе. Все мержи — вручную пользователем.

## 12. Что Claude делает в начале каждой сессии в `gold-bot/`

1. Читает `gold-bot/CLAUDE.md` (этот файл).
2. Читает `gold-bot/risk-profile.md` если работает с числами.
3. Читает соответствующий план в `gold-bot/plans/` если работа по плану.
4. **Не читает** файлы корня репозитория (`/CLAUDE.md`, `/AGENTS.md`, `/бизнес/`) — они от другого проекта.
5. Если вопрос неясен — спрашивает, не выдумывает.

## 13. Что Claude НЕ делает без явного разрешения

- Не меняет `gold-bot/CLAUDE.md`, `gold-bot/risk-profile.md` (только diff + обоснование, потом одобрение).
- Не трогает корневые файлы репозитория.
- Не запускает live-торговлю.
- Не пушит в `main`.
- Не создаёт API-ключи, не хранит секреты в коде или планах.
- Не торгует реальными деньгами выше суммы, явно разрешённой текущей фазой (см. `risk-profile.md` → «Фазы капитала»).
