## 1. Первичный замысел

Формализовать **композитную стратегию** в два сетапа и довести до бэктеста
на **BTC** (только BTC в Ф1; SOL/золото/серебро — позже, отдельными фазами).

- **Сетап A — Liquidation Reversal** (A1 LONG / A2 SHORT): разворот от
  каскада ликвидаций на значимом экстремуме, с подтверждением OI + CVD.
- **Сетап B — OI-Breakout из боковика**: ранний вход в тренд по резкому
  росту открытого интереса из низкой базы синхронно с ценой.

Метрики: ликвидации (триггер), Open Interest (главный gate направления),
CVD/дельта (подтверждение/дивергенция), фандинг (анти-фильтр).

## 4. Адекватизация (стыковка с правилами бота «трейдер»)

Источник истины — `трейдер/правила/`. Стыковка с курсом:

| Параметр | Курс Щукина | Правила «трейдер» | Решение |
|---|---|---|---|
| Риск/сделку | 0.5–1% | **0.5% ($50)** | берём правило бота |
| R:R | старт 1:3, мин 1:2 | мин **1.5R** (пол) | цель сетапа 1:3, фильтр ≥1.5R соблюдён |
| Размер позиции | риск / дист. стопа | то же (от риска) | совпадает |
| Плечо | «нерелевантно» | потолки 5x / серебро 3x | берём правило бота |
| Стоп | структурный за экстремум | биржевой вместе с входом | структурный SL = биржевой ордер |
| Дневной стоп | не задан | **−4%/день** | берём правило бота |
| ≤2 позиции, без корреляции | — | правило бота | в Ф1 не актуально (только BTC) |

**Берём из курса:** liquidation-reversal + OI-breakout + CVD/funding
фильтры + структурный SL + цель 1:3.
**Отбрасываем (конспект `что-отбросить.md`):** усреднение убытка,
хедж-убыточной, ручные правки позиции, весь маркетинг. Фирменный
бренд-акроним не используем (терминологическое правило).

### Решение Р1 — РЕШЕНО: сравнить бэктестом (2026-05-18)

Тестируем **оба** варианта сетапа A на BTC и выбираем по цифрам:
- **Swing-вариант:** структурный SL за экстремум, цель 1:3, удержание часы.
- **HF-скальп-вариант:** тейки 2–6%, отработка 10–15 мин.
Сравнение на одних и тех же сигналах/периоде (см. фазу Ф1.4). Критерий
выбора: лучший по PF×Sharpe при ≥30 сделок OOS И с учётом комиссий/
проскальзывания (HF-вариант штрафуется реалистичной моделью costs).

## 5. Десять причин провала (критика до кода)

1. Нет OI-фида (B1) → половина логики не воспроизводима, шорт-gate не
   работает.
2. Нет истории ликвидаций (B2) → бэктест на проксе недостоверен.
3. Overfitting порогов ликвидаций ($100k мажоры и т.п.) под BTC-историю.
4. Скальп-режим (Р1) на автономе → латенси/комиссии съедают edge
   (ровно сценарий, который чуть не слил депозит на серебре ранее).
5. «Значимый экстремум» дискреционен — формализация (Donchian-N?) может
   не совпасть с тем, что имел в виду курс → ложные сигналы.
6. «Цикл ликвидаций завершён» без чёткого критерия → ранний вход в
   падающий нож.
7. Пороги ликвидаций абсолютные ($) — не масштабируются по режиму
   волатильности/времени.
8. CVD-дивергенция шумна на низких ТФ → много ложных подтверждений.
9. Survivorship/look-ahead в данных ликвидаций (агрегатор пересчитывает
   историю).
10. Малая выборка сделок на одном BTC → статистически незначимо
    (<30 сделок OOS → не принимать).

### ⚠️ Дисклеймер Claude (durable, не удалять)

Это **разведочный экран для отсева, НЕ валидация edge**. Риски, принятые
осознанно пользователем:
- 9 сделок статистически незначимо; PF/Sharpe на такой выборке —
  шум с широким доверительным интервалом.
- Прогон планки по многим монетам = multiple-testing / selection bias:
  часть пройдёт случайно.
- Нет OI-гейта (ядро методики), окно ~7 мес вместо ≥2 лет.

**Жёсткое правило:** ничто, прошедшее этот экран, НЕ идёт в testnet/live
без полноценной валидации (≥2 года, ≥30 сделок OOS, с OI-гейтом —
критерии §6/§7). Экран только сужает список кандидатов на полную
проверку, не заменяет её.

## 6c. Корректировки 2026-05-19 (платный ключ, разбор)

Пользователь уточнил: ключ платный. Перепроверка вскрыла ДВЕ мои
ранние ошибки + один жёсткий потолок:

1. **OI-история доступна.** Клиент дёргал `/open-interest/aggregated-

2. **Ликвидации по альтам ЕСТЬ.** XRP/SOL/DOGE/… отдают 1000 точек.
   Ярлык «Н/Д — нет данных» в экране был НЕВЕРЕН: ноль сделок = я
   применил к альтам BTC-порог `liq_min_baseline_usd=100000` (для
   альтов по методике $15–30k) + короткое окно. Это конфиг, не данные.
3. **Жёсткий потолок истории (неподвижный).** API: «earliest allowed
   start_time = 2025-11-20». Тариф отдаёт 4h-историю только за ~6 мес,
   пагинация глубже обрезается. 1d даёт ~2.7 года, но дневной ТФ для
   разворотной стратегии не годится.

**Следствие.** BTC с OI-гейтом на ~7 мес = **4 сделки** (без гейта 9),
Sharpe −1.34. На ≤6 мес 4h любой символ даёт 1-значное число сделок —
это шум, не edge, при любом OI и по любому списку. Прогон «минимум по
многим монетам» на этом окне = data-dredging по шуму. Полная валидация
(≥2 года, ≥30 сделок) на этом ключе НЕВОЗМОЖНА — нужен тариф с большей
глубиной 4h-истории, либо forward-paper-сбор.

## 6f. Улучшение №1 — TF-масштабированный стоп (логика, не подгон)

Диагноз 6h-прогона: 3 из 4 сделок вышли по STOP_LOSS в **ту же свечу**
(длительность 0 мин), дистанция стопа ~0.6–0.7%. Причина: структурный
стоп ставится вплотную за экстремум, а floor `stop_min_pct=0.5%` —
масштаб 5m. На 6h-баре BTC внутрибаровый шум 1–3% мгновенно выбивает.

Это **мис-спека**, не «плохая стратегия» и не повод для подгона:
- `трейдер/правила/риск-профиль.md`: дистанция стопа — от волатильности
  (ATR), не фикс-%. Текущая стратегия ATR не использует — расхождение
  правил бота и переиспользуемого кода.
- `конспект/правила-выхода.md` (методичка): SL зависит от ТФ —
  5m→0.4%, 30m→1.2%, 4h→4.4%. Floor должен масштабироваться по ТФ.

Источник числа — таблица методички, НЕ максимизация Sharpe (AGENTS.md).
TF→`stop_min_pct` (из таблицы методички, экстраполяция вверх):

| ТФ | stop_min_pct |
|---|---|
| 5m | 0.4 |
| 15m | 0.8 |
| 1h | 2.0 |
| 4h | 4.4 |
| 6h | 5.0 |
| 8h | 6.0 |
| 12h | 7.0 |

**Фаза Ф1.3a (план перед кодом):**
1. `композит-btc-6h.yaml`: `stop_min_pct: 5.0` (из таблицы методички).
2. Перепрогон BTC 6h — проверка, что патология «стоп в 0 минут»
